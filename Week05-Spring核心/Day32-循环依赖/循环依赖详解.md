# Day32 详解 — 循环依赖与@Transactional原理

## 〇、Spring三级缓存版本演进

- **三级缓存机制是Spring独有的**，非JSR标准，是IoC容器的内部实现细节
- **Spring 6.x**：三级缓存机制核心未变，但解决循环依赖限制与5.x相同（仅支持单例Bean的Setter注入）
- **Spring Boot 3.x**：底层基于Spring 6.x，`@Transactional`代理机制不变（CGLIB），循环依赖相关行为不变
- **重要限制**：构造器注入无法解决循环依赖（所有版本均如此，因为对象尚未创建完成）

```java
// Spring 6.x / Boot 3.x 版本信息
// spring-beans-6.x → DefaultSingletonBeanRegistry 中三级缓存定义
public class DefaultSingletonBeanRegistry extends SimpleAliasRegistry implements SingletonBeanRegistry {
    /** 一级缓存：完全初始化的Bean */
    private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);
    /** 二级缓存：提前曝光的Bean实例（未完成属性填充） */
    private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);
    /** 三级缓存：Bean工厂（用于生成提前引用的代理对象） */
    private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);
}
```

## 一、什么是循环依赖

### 1.1 定义

**循环依赖**是指两个或多个Bean之间相互持有对方的引用，形成了一个闭环引用链。

```
最简单形式：
    ClassA ──依赖──> ClassB
      ^               │
      │               │
      └──依赖─────────┘

更多参与者：
    A → B → C → A  （环形依赖链）
```

对于Spring容器，循环依赖的**本质问题**是：在创建A时需要注入B，而创建B时又需要注入A，此时A和B都未创建完成，陷入"鸡生蛋、蛋生鸡"的困境。

### 1.2 常见场景

| 场景 | 能否解决 | 说明 |
|------|---------|------|
| **属性注入循环依赖** | ✅ 可解决 | `@Autowired` 字段注入，Spring通过三级缓存自动解决 |
| **Setter注入循环依赖** | ✅ 可解决 | `@Autowired` setter方法，同样通过三级缓存解决 |
| **构造器注入循环依赖** | ❌ 无法解决 | 抛出 `BeanCurrentlyInCreationException` |
| **prototype作用域** | ❌ 无法解决 | 原型Bean不经过缓存，无法提前曝光 |

```java
// ✅ Setter注入 - 可解决
@Component
public class ServiceA {
    private ServiceB serviceB;

    @Autowired
    public void setServiceB(ServiceB serviceB) {
        this.serviceB = serviceB;
    }
}

@Component
public class ServiceB {
    private ServiceA serviceA;

    @Autowired
    public void setServiceA(ServiceA serviceA) {
        this.serviceA = serviceA;
    }
}
```

```java
// ❌ 构造器注入 - 无法解决，抛出异常
@Component
public class ServiceA {
    private final ServiceB serviceB;

    public ServiceA(ServiceB serviceB) { // 需要ServiceB实例
        this.serviceB = serviceB;
    }
}

@Component
public class ServiceB {
    private final ServiceA serviceA;

    public ServiceB(ServiceA serviceA) { // 需要ServiceA实例 → 死循环！
        this.serviceA = serviceA;
    }
}
// 抛出: BeanCurrentlyInCreationException: Error creating bean with name 'serviceA':
// Requested bean is currently in creation: Is there an unresolvable circular reference?
```

## 二、三级缓存结构

### 2.1 三级缓存定义

Spring在 `DefaultSingletonBeanRegistry` 中定义了三个Map作为缓存：

```java
public class DefaultSingletonBeanRegistry extends SimpleAliasRegistry implements SingletonBeanRegistry {

    /** 一级缓存（singletonObjects）
     *  存放完全初始化完成的单例Bean
     *  Key: beanName, Value: 完全就绪的Bean实例（包括代理对象） */
    private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

    /** 二级缓存（earlySingletonObjects）
     *  存放提前曝光的Bean实例（原始对象或代理对象，但属性未填充完成）
     *  Key: beanName, Value: 未完成属性填充的Bean实例 */
    private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);

    /** 三级缓存（singletonFactories）
     *  存放ObjectFactory，用于延迟生成Bean的代理对象
     *  Key: beanName, Value: 可以生成该Bean提前引用的工厂 */
    private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);

    /** 正在创建中的Bean名称集合（用于检测循环依赖） */
    private final Set<String> singletonsCurrentlyInCreation =
            Collections.newSetFromMap(new ConcurrentHashMap<>(16));
}
```

### 2.2 ASCII结构图

```
Bean创建时在三级缓存中的流转过程：

创建开始
   │
   ▼
┌─────────────────────────────────────────────────────────────┐
│  Step1: 实例化（调用构造器）                                  │
│          ↓                                                  │
│  Step2: 放入三级缓存 singletonFactories                       │
│          ObjectFactory { getObject() { return getEarlyBeanRef() } }
│          ↓                                                  │
│  Step3: 属性填充（populateBean）                              │
│          ↓  （如果需要依赖其他Bean，会触发依赖Bean的创建）      │
│  Step4: 初始化（initializeBean）                             │
│          ↓  （执行 Aware、PostProcessor、init-method...）     │
│  Step5: 放入一级缓存 singletonObjects                         │
│          并从二/三级缓存中移除                                 │
└─────────────────────────────────────────────────────────────┘



三级缓存协作关系：

    singletonObjects（一级）          earlySingletonObjects（二级）    singletonFactories（三级）
   ┌──────────────────────┐        ┌──────────────────────┐        ┌──────────────────────┐
   │ 完全初始化完成的Bean    │        │ 提前曝光的半成品Bean    │        │ ObjectFactory(代理工厂) │
   │                      │        │ （已从三级升级到二级）   │        │ 可以生成提前引用        │
   │ getBean("a")         │        │                      │        │                      │
   │ → 直接返回           │        │ getSingleton()        │        │ getObject() →        │
   │                      │  从三级┌── 先从二级取           │   存入  │   1. getEarlyBeanRef  │
   │                      │  搬来  │                      │  二级   │   2. 可能创建代理对象  │
   └──────────────────────┘        └──────────────────────┘        └──────────────────────┘
         最终归宿                       中间暂存                      最早存放
```

### 2.3 各缓存作用对比表

| 缓存 | 名称 | 存放内容 | 何时放入 | 何时移除 | 作用 |
|------|------|---------|---------|---------|------|
| **一级** | `singletonObjects` | 完全初始化的Bean | `initializeBean`完成后 | 容器销毁时 | 对外提供就绪的Bean |
| **二级** | `earlySingletonObjects` | 提前曝光的半成品Bean | 从三级缓存升级而来 | 初始化完成移到一级 | 确保同一循环依赖中的Bean引用一致 |
| **三级** | `singletonFactories` | `ObjectFactory` | 实例化后、属性填充前 | 被调用后移到二级 | 延迟生成代理对象（AOP切面） |

## 三、循环依赖解决流程

### 3.1 正常创建流程回顾

```
正常Bean创建流程（无循环依赖时）：

getBean("a")
  → getSingleton("a")               // 从一级缓存查找 → 未找到
  → createBean("a")
      → doCreateBean("a")
          → createBeanInstance()    // 1.实例化（调用构造器 → 原始对象a）
          → addSingletonFactory()   // 2.放入三级缓存
          → populateBean()          // 3.属性填充（getBean("b") → 发现b不存在 → 创建b）
          → initializeBean()        // 4.初始化（aware → postProcess → initMethod）
  → addSingleton("a")              // 5.放入一级缓存，清二/三级
```

### 3.3 核心代码位置

**DefaultSingletonBeanRegistry** 中的关键方法：

```java
// 获取单例Bean的核心方法（Spring源码简化版）
protected Object getSingleton(String beanName, boolean allowEarlyReference) {
    // 第一步：从一级缓存取
    Object singletonObject = this.singletonObjects.get(beanName);
    // 如果一级没有，且该Bean正在创建中 → 说明可能有循环依赖
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
        // 第二步：从二级缓存取（提前曝光的半成品）
        singletonObject = this.earlySingletonObjects.get(beanName);
        if (singletonObject == null && allowEarlyReference) {
            synchronized (this.singletonObjects) {
                // 双重检查
                singletonObject = this.singletonObjects.get(beanName);
                if (singletonObject == null) {
                    singletonObject = this.earlySingletonObjects.get(beanName);
                    if (singletonObject == null) {
                        // 第三步：从三级缓存取ObjectFactory
                        ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
                        if (singletonFactory != null) {
                            // 调用工厂方法生成提前引用（可能创建代理）
                            singletonObject = singletonFactory.getObject();
                            // 升级到二级缓存，从三级移除
                            this.earlySingletonObjects.put(beanName, singletonObject);
                            this.singletonFactories.remove(beanName);
                        }
                    }
                }
            }
        }
    }
    return singletonObject;
}
```

```java
// 将Bean的ObjectFactory放入三级缓存（实例化之后、属性填充之前调用）
protected void addSingletonFactory(String beanName, ObjectFactory<?> singletonFactory) {
    Assert.notNull(singletonFactory, "Singleton factory must not be null");
    synchronized (this.singletonObjects) {
        if (!this.singletonObjects.containsKey(beanName)) {
            this.singletonFactories.put(beanName, singletonFactory);  // 放入三级
            this.earlySingletonObjects.remove(beanName);               // 清理二级（如有）
            this.registeredSingletons.add(beanName);
        }
    }
}
```

## 四、为什么需要三级缓存

### 4.1 两级缓存能解决什么问题

如果只有**一级缓存**（singletonObjects）和**二级缓存**（earlySingletonObjects），能否解决循环依赖？

```
两级缓存方案：
  getBean("A") → 实例化 → 放入二级缓存（存原始对象）
  → 填充属性需要B → 创建B → B填充属性需要A
  → 从二级缓存取到A的原始对象 → 完成注入
  → B初始化 → 放一级 → A填充完成 → A初始化 → 放一级

结论：对于普通Bean（无AOP代理），两级缓存足够。
```

### 4.2 为什么还需要第三级

**核心原因**：Spring AOP代理对象需要在Bean初始化完成后才正式创建，但循环依赖可能要求在属性填充阶段就拿到代理对象。

```
问题场景：
  ServiceA 被 @Transactional 标记（需要AOP代理）
  ServiceB 依赖 ServiceA

如果只有两级缓存（存原始对象）：
  getBean("A") → 实例化 → 二级缓存存入原始对象a(未代理)
  → B填充时需要A → 从二级拿到原始对象a
  → B注入的是原始对象a（不是代理对象！）
  → A初始化完成 → 后置处理器创建代理对象 proxyA
  → B中持有的是原始对象a ❌ 而Spring容器返回的是proxyA ❌ 不一致！

如果Spring在放入二级缓存时就创建代理：
  getBean("A") → 实例化 → 立即创建代理 → 二级缓存存入proxyA
  → 但是A此时属性还未填充，创建代理为时过早 ⚠️
  → 可能影响其他后置处理器的正常执行
```

**三级缓存的精妙之处**：

```java
// 三级缓存的ObjectFactory：
// 延迟到真正被引用时才决定返回原始对象还是代理对象
singletonFactories.put("serviceA", () -> {
    // 此时Spring可以判断是否需要为A创建代理
    // 如果需要AOP → 返回代理对象
    // 如果不需要 → 返回原始对象
    return getEarlyBeanReference(beanName, mbd, bean);
});
```

### 4.3 详细解释

```
三级缓存解决AOP代理问题的完整流程：

ServiceA（有@Transactional） ←→ ServiceB（无代理）

getBean("serviceA")
  → 实例化ServiceA → 原始对象a₀
  → 放入三级缓存：
      singletonFactories["serviceA"] = () → {
          // SmartInstantiationAwareBeanPostProcessor.getEarlyBeanReference()
          // 检查是否需要AOP → 需要则创建代理
          return wrapIfNecessary(a₀); // 返回proxyA
      }
  → 属性填充：需要serviceB
      └→ getBean("serviceB")
          → 实例化ServiceB → b₀
          → 属性填充：需要serviceA
              └→ getSingleton("serviceA")
                  → 三级缓存命中！调用getObject()
                  → 返回 proxyA ✅（提前创建的代理对象）
                  → 升级到二级缓存
          → b₀注入proxyA → b₀初始化完成 → 放一级
  → a₀拿到b₀引用 → a₀属性填充完成
  → a₀初始化 → 后置处理器再次尝试创建代理
      → 发现二级缓存中已有proxyA → 直接复用，不再重复创建
  → 将proxyA放入一级缓存
  → 清空二/三级

最终结果：一级缓存中的serviceA == 二级缓存中的serviceA == B注入的serviceA
         都是同一个代理对象 ✅
```

**关键设计**：如果二级缓存中已经有该Bean，说明提前引用已经被使用过，后续初始化阶段就不再创建新的代理，直接复用已有的代理对象，保证引用一致性。

## 五、@Lazy解决循环依赖

### 5.1 @Lazy原理

`@Lazy` 注解告诉Spring容器**延迟初始化**该Bean：不在容器启动时创建，而是在**第一次调用时**才创建。注入的不是真正的Bean实例，而是一个**代理对象**。

```
@Lazy解决循环依赖的本质：

A → B  （A依赖B）
B → A  （B依赖A，但A被@Lazy标记）

注入到B中的A其实是一个代理对象（懒加载代理）。
当B真正调用A的方法时，代理才去Spring容器中获取真正的A实例。
此时A可能已经创建完成。
```

### 5.2 使用方式

```java
// 方式一：属性注入 + @Lazy
@Component
public class ServiceA {
    @Autowired
    private ServiceB serviceB;
}

@Component
public class ServiceB {
    @Autowired
    @Lazy  // 注入的是懒加载代理，不是真正的ServiceA实例
    private ServiceA serviceA;

    public void doSomething() {
        // 真正调用方法时，代理才从容器获取ServiceA实例
        serviceA.execute();
    }
}
```

```java
// 方式二：构造器注入 + @Lazy（构造器循环依赖的唯一解法）
@Component
public class ServiceA {
    private final ServiceB serviceB;

    public ServiceA(ServiceB serviceB) {
        this.serviceB = serviceB;
    }
}

@Component
public class ServiceB {
    private final ServiceA serviceA;

    public ServiceB(@Lazy ServiceA serviceA) { // @Lazy让Spring注入代理而非真实对象
        this.serviceA = serviceA;
    }
}
```

```java
// 方式三：@Configuration类中使用@Lazy
@Configuration
public class AppConfig {
    @Bean
    @Lazy
    public ServiceA serviceA() {
        return new ServiceA();
    }

    @Bean
    public ServiceB serviceB(ServiceA serviceA) {
        return new ServiceB(serviceA); // 注入的是懒加载代理
    }
}
```

### 5.3 @Lazy的局限性

| 限制 | 说明 |
|------|------|
| **首次调用有性能开销** | 第一次调用时会触发Bean的完整创建流程 |
| **可能隐藏真正的循环依赖** | 如果循环依赖不存在，@Lazy只是推迟了创建，不是真正解决问题 |
| **调试困难** | 异常堆栈中出现的是代理类，定位问题较复杂 |
| **不适用于需要立即初始化的Bean** | 如需要在启动时校验的Bean |
| **序列化问题** | 懒加载代理对象序列化可能出问题 |

## 六、@Transactional注解原理

### 6.1 核心原理：AOP + TransactionManager

`@Transactional` 是Spring声明式事务的核心注解，底层基于 **AOP代理** + **事务管理器** 实现。

```
@Transactional 工作流程：

   调用代理对象的方法
         │
         ▼
   ┌──────────────────┐
   │  TransactionInterceptor  │   // AOP环绕通知
   │  (MethodInterceptor)     │
   └──────────┬───────────────┘
              │
              ▼
   ┌──────────────────┐
   │ 1. 开启事务        │   // PlatformTransactionManager.getTransaction()
   └──────────┬─────────┘
              ▼
   ┌──────────────────┐
   │ 2. 执行目标方法     │   // invocation.proceed() → 调用真实业务方法
   └──────────┬─────────┘
              ▼
   ┌──────────────────┐
   │ 3. 提交/回滚事务   │   // 根据异常类型决定 commit() 或 rollback()
   └──────────────────┘
```

### 6.2 @Transactional与AOP的关系

```java
// Spring启动时的处理流程：

// 1. 扫描@Transactional注解
// 2. 创建Advisor（BeanFactoryTransactionAttributeSourceAdvisor）
// 3. 为匹配的Bean创建AOP代理（CGLIB，Spring Boot默认proxyTargetClass=true）
// 4. 代理对象中包含TransactionInterceptor

// TransactionInterceptor核心逻辑（简化版）
public class TransactionInterceptor implements MethodInterceptor {
    @Override
    public Object invoke(MethodInvocation invocation) throws Throwable {
        // 获取事务属性（传播行为、隔离级别、超时等）
        TransactionAttribute attr = getTransactionAttribute(invocation.getMethod());

        // 获取事务管理器
        PlatformTransactionManager tm = determineTransactionManager(attr);

        // 开启事务（根据传播行为决定是新建还是加入已有事务）
        TransactionStatus status = tm.getTransaction(attr);

        Object retVal;
        try {
            // 执行目标方法
            retVal = invocation.proceed();
        } catch (Throwable ex) {
            // 根据异常类型决定是否回滚
            completeTransactionAfterThrowing(attr, status, ex);
            throw ex;
        }

        // 正常返回 → 提交事务
        tm.commit(status);
        return retVal;
    }
}
```

### 6.3 事务代理创建过程

```
Bean创建过程中事务代理的创建时机：

createBean("userService")
  → doCreateBean()
      → createBeanInstance()        // 实例化 → 原始对象
      → addSingletonFactory()       // 放入三级缓存（此时无代理）
      → populateBean()              // 属性填充
      → initializeBean()            // 初始化
          → applyBeanPostProcessorsAfterInitialization()
              → AbstractAutoProxyCreator.postProcessAfterInitialization()
                  → wrapIfNecessary()  // 创建事务代理对象！✅
                  → 返回 CGLIB 代理对象（非原始对象）
      → 最终放入一级缓存的是代理对象

注意：代理创建发生在 initializeBean() 之后！
     如果存在循环依赖，三级缓存的 getEarlyBeanReference() 
     也会被调用，提前创建代理。
```

## 七、@Transactional失效场景

### 7.1 自调用失效（最常见）

```java
@Service
public class UserService {

    @Transactional
    public void methodA() {
        // 事务方法
        jdbcTemplate.update("INSERT INTO users VALUES (?)", "张三");

        // 直接调用同类中的另一个事务方法 → 事务失效 ❌
        this.methodB();  // this 是原始对象，不是代理对象！
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() {
        // 期望开启新事务，但实际上 methodB的事务注解完全无效
        jdbcTemplate.update("INSERT INTO orders VALUES (?)", "订单001");
    }
}
```

**原因**：`this` 指向的是原始对象，不是Spring生成的代理对象，因此AOP拦截不到调用。

```
自调用失效原理：

   外部调用者
       │
       ▼
   ┌─────────┐   调用 methodA()     ┌──────────────┐
   │ 调用者    │ ──────────────────→ │ 代理对象       │
   └─────────┘                      │ (CGLIB Proxy) │
                                    └──────┬───────┘
                                           │ 开启事务 ✅
                                           ▼
                                    ┌──────────────┐
                                    │ 原始对象(this) │
                                    │              │
                                    │ this.methodB()│  ← 直接调用，绕过代理！
                                    │ 事务注解失效 ❌  │
                                    └──────────────┘
```

**解决方案：**

```java
// 方案一：通过AopContext获取当前代理对象
@Service
public class UserService {

    @Transactional
    public void methodA() {
        jdbcTemplate.update("INSERT INTO users VALUES (?)", "张三");

        // 获取当前代理对象，通过代理调用
        UserService proxy = (UserService) AopContext.currentProxy();
        proxy.methodB();  // ✅ methodB的事务生效
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() {
        jdbcTemplate.update("INSERT INTO orders VALUES (?)", "订单001");
    }
}

// 需要在配置类或启动类上开启 exposeProxy
// @EnableAspectJAutoProxy(exposeProxy = true)
```

```java
// 方案二：将methodB抽取到另一个Service中
@Service
public class OrderService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() {
        jdbcTemplate.update("INSERT INTO orders VALUES (?)", "订单001");
    }
}

@Service
public class UserService {
    @Autowired
    private OrderService orderService;

    @Transactional
    public void methodA() {
        jdbcTemplate.update("INSERT INTO users VALUES (?)", "张三");
        orderService.methodB();  // ✅ 通过注入的代理对象调用，事务生效
    }
}
```

```java
// 方案三：注入自身
@Service
public class UserService {
    @Autowired
    private UserService self;  // Spring注入的是代理对象

    @Transactional
    public void methodA() {
        jdbcTemplate.update("INSERT INTO users VALUES (?)", "张三");
        self.methodB();  // ✅ 通过注入的代理调用，事务生效
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() {
        jdbcTemplate.update("INSERT INTO orders VALUES (?)", "订单001");
    }
}
```

### 7.2 非public方法

```java
@Service
public class UserService {

    // CGLIB代理可以拦截protected和包可见方法
    // JDK动态代理只能拦截public方法
    // 但Spring官方规范要求必须public

    @Transactional
    private void doSomething() {  // ❌ private方法，事务不生效
        // ...
    }

    @Transactional
    protected void doProtected() {  // ⚠️ CGLIB下可生效，但不推荐
        // ...
    }

    @Transactional
    public void doPublic() {  // ✅ 推荐做法
        // ...
    }
}
```

### 7.3 异常被catch吞掉

```java
@Service
public class UserService {

    @Transactional
    public void saveUser() {
        try {
            jdbcTemplate.update("INSERT INTO users VALUES (?)", "张三");
            int result = 1 / 0;  // 抛出ArithmeticException
        } catch (Exception e) {
            // ❌ 异常被catch吞掉了，Spring感知不到异常，事务不会回滚
            log.error("保存失败", e);
            // 没有重新抛出异常！
        }
        // 事务正常提交 ❌ 数据被错误地持久化了
    }
}
```

```java
// ✅ 正确做法
@Transactional(rollbackFor = Exception.class)  // 明确指定回滚异常类型
public void saveUser() {
    try {
        jdbcTemplate.update("INSERT INTO users VALUES (?)", "张三");
        int result = 1 / 0;
    } catch (Exception e) {
        log.error("保存失败", e);
        // 方案一：重新抛出运行时异常
        throw new RuntimeException("保存用户失败", e);
        // 方案二：手动回滚事务
        // TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
        throw e;  // 必须抛出异常才能触发回滚
    }
}
```

### 7.4 异常类型不匹配

```java
@Service
public class UserService {

    @Transactional  // 默认只回滚 RuntimeException 和 Error
    public void saveUser() throws Exception {
        jdbcTemplate.update("INSERT INTO users VALUES (?)", "张三");
        throw new Exception("业务异常");  // ❌ checked异常，默认不回滚！
    }
}
```

```java
// ✅ 正确配置
@Transactional(rollbackFor = Exception.class)  // 指定所有异常都回滚
public void saveUser() throws Exception {
    jdbcTemplate.update("INSERT INTO users VALUES (?)", "张三");
    throw new Exception("业务异常");  // ✅ 现在会回滚
}
```

| rollbackFor配置 | 回滚范围 |
|----------------|---------|
| **默认（不配置）** | RuntimeException + Error |
| `rollbackFor = Exception.class` | RuntimeException + Checked Exception |
| `rollbackFor = {SQLException.class, IOException.class}` | 指定具体异常 |
| `noRollbackFor = IllegalArgumentException.class` | 排除某些异常不回滚 |

### 7.5 数据库引擎不支持事务

```sql
-- MyISAM引擎不支持事务，@Transactional不生效
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(100)
) ENGINE = MyISAM;  -- ❌ 非事务引擎

-- 应使用InnoDB
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(100)
) ENGINE = InnoDB;  -- ✅ 支持事务
```

### 7.6 多线程调用

```java
@Service
public class UserService {

    @Transactional
    public void batchSave() {
        // 主线程中有事务

        new Thread(() -> {
            // ❌ 新线程中，事务不会传播过来
            // 每个线程有独立的数据库连接，独立的事务
            jdbcTemplate.update("INSERT INTO users VALUES (?)", "李四");
        }).start();

        // 子线程中的操作与主线程事务无关
    }
}
```

### 7.7 @Transactional方法被子类重写

```java
// 父类
@Service
public class BaseService {

    @Transactional
    public void save() {
        jdbcTemplate.update("...");
        doExtra();  // 调用了子类重写的方法
    }

    protected void doExtra() {
        // 父类空实现
    }
}

// 子类
@Service
public class ChildService extends BaseService {

    @Override
    public void save() {  // ⚠️ 重写了父类的@Transactional方法
        // 子类的save方法上没有@Transactional，事务失效！
        super.save();
    }

    @Override
    protected void doExtra() {
        // 这个方法在父类Transactional方法内被调用，有事务
    }
}
```

### 失效场景总结表

| 失效场景 | 原因 | 解决方案 |
|---------|------|---------|
| **自调用** | `this.method()` 绕过代理 | `AopContext.currentProxy()` / 注入自身 |
| **非public方法** | AOP无法拦截 | 改为public |
| **异常被catch** | Spring感知不到异常 | 重新抛异常或手动setRollbackOnly |
| **异常类型不匹配** | 默认只回滚RuntimeException | 配置rollbackFor |
| **非事务引擎** | MyISAM等不支持事务 | 使用InnoDB |
| **多线程** | 事务绑定在线程上 | 线程间独立管理事务 |
| **子类重写** | 代理方法被覆盖 | 确保代理方法最终被调用 |

## 八、面试常见问题

**Q1：Spring如何解决循环依赖？**

A：Spring通过**三级缓存**机制解决单例Bean的Setter注入循环依赖：
1. `singletonObjects`（一级）存放完全初始化的Bean
2. `earlySingletonObjects`（二级）存放提前曝光的半成品Bean
3. `singletonFactories`（三级）存放ObjectFactory用于延迟生成代理对象

核心流程：A实例化后放入三级缓存 → A填充属性时发现需要B → B实例化后需要A → 从三级缓存获取A的提前引用（可能创建代理） → B完成创建 → A完成创建。

---

**Q2：为什么构造器注入无法解决循环依赖？**

A：构造器注入时，对象尚未实例化完成，Bean引用也还未放入任何缓存。当A的构造器需要B，Spring去创建B，B的构造器又需要A，此时A还未完成构造器执行，不在任何缓存中，因此形成死循环，抛出 `BeanCurrentlyInCreationException`。

---

**Q3：三级缓存能否简化为两级？**

A：理论上，如果项目中**没有AOP代理**（没有`@Transactional`、`@Async`等），两级缓存（一级+二级）足够。但Spring需要支持AOP场景：代理对象应该在初始化阶段由后置处理器创建，而循环依赖可能要求在初始化完成前就暴露引用。三级缓存的`ObjectFactory`可以**延迟到真正被引用时才决定**返回原始对象还是代理对象，这是两级缓存无法做到的。

---

**Q4：@Lazy与三级缓存解决循环依赖有什么区别？**

A：
- **三级缓存**：通过提前曝光半成品解决，Spring自动处理，但仅支持Setter注入的单例Bean
- **@Lazy**：通过注入代理对象延迟初始化，打破创建依赖链，支持构造器注入和所有注入方式
- `@Lazy`更通用但首次调用有性能开销；三级缓存是默认机制，零配置

---

**Q5：@Transactional为什么不生效？**

A：常见原因按频率排序：
1. **自调用**（`this.xxx()`）——调用没有经过代理对象
2. **异常被catch吞掉**——Spring感知不到异常
3. **方法非public**——AOP无法拦截
4. **异常类型不匹配**——默认只回滚RuntimeException，而抛出了checked异常
5. **数据库引擎不支持事务**——如MyISAM

---

**Q6：三级缓存中的ObjectFactory什么时候被调用？**

A：在 `getSingleton(String beanName, boolean allowEarlyReference)` 方法中。当：
1. 一级缓存中没有该Bean
2. 该Bean正在创建中（`isSingletonCurrentlyInCreation`返回true）
3. 二级缓存中也没有
4. `allowEarlyReference` 为 true

此时从三级缓存取出ObjectFactory，调用 `getObject()` 方法。调用完成后，结果放入二级缓存，并从三级缓存中移除。

---

**Q7：为什么Spring需要二级缓存？直接从三级取不行吗？**

A：二级缓存的作用是**确保同一Bean的提前引用在多处使用时保持一致**。假如B和C都依赖A，它们都会触发三级缓存的ObjectFactory调用。如果ObjectFactory每次调用都创建新代理，那B和C拿到的A引用就不同了。二级缓存将ObjectFactory的结果缓存下来，确保同一个Bean的提前引用全局唯一。

## 九、LeetCode题目解析

### 9.1 [33. 搜索旋转排序数组（Medium）](https://leetcode.cn/problems/search-in-rotated-sorted-array/)

**题目描述**：整数数组 `nums` 原本升序排列，在某个未知下标 `k` 处旋转（`0 <= k < nums.length`）。给定旋转后的数组和一个目标值 `target`，返回目标值的下标，不存在返回 -1。要求 O(log n) 时间复杂度。

**思路分析**：二分查找的变体。旋转后的数组**部分有序**——将数组从 `mid` 处一分为二，至少有一半是有序的。判断 target 是否在有序的一半中，从而缩小搜索范围。

```
旋转数组示意：
        ┌─ 递增段1
        │        ┌─ 递增段2
        ▼        ▼
    [4, 5, 6, 7, 0, 1, 2]
            ↑        ↑
         旋转点     最小值

mid 将数组分为两半，至少一半有序：
  - [left, mid] 有序 → 判断target是否在其中
  - [mid, right] 有序 → 判断target是否在其中
```

```java
public class SearchRotatedArray {
    public int search(int[] nums, int target) {
        if (nums == null || nums.length == 0) return -1;

        int left = 0, right = nums.length - 1;

        while (left <= right) {
            int mid = left + (right - left) / 2;
            if (nums[mid] == target) return mid;

            // 判断哪一半是有序的
            if (nums[left] <= nums[mid]) { // 左半部分有序
                if (nums[left] <= target && target < nums[mid]) {
                    right = mid - 1; // target在左半部分
                } else {
                    left = mid + 1;  // target在右半部分
                }
            } else { // 右半部分有序
                if (nums[mid] < target && target <= nums[right]) {
                    left = mid + 1;  // target在右半部分
                } else {
                    right = mid - 1; // target在左半部分
                }
            }
        }
        return -1;
    }
}
```

**复杂度分析**：
- 时间复杂度：O(log n)，标准二分查找
- 空间复杂度：O(1)，原地查找

### 9.2 [153. 寻找旋转排序数组中的最小值（Medium）](https://leetcode.cn/problems/find-minimum-in-rotated-sorted-array/)

**题目描述**：给定一个元素互不相同的旋转排序数组，找出其中的最小值。要求 O(log n) 时间复杂度。

**思路分析**：旋转后，最小值就是**旋转点**。同样使用二分查找，比较 `nums[mid]` 和 `nums[right]` 来判断最小值在哪一侧。

```
关键洞察：
  - 如果 nums[mid] > nums[right] → 最小值在 [mid+1, right]
  - 如果 nums[mid] < nums[right] → 最小值在 [left, mid]

为什么不是 nums[mid] < nums[left]？
因为旋转后left可能指向较大值，用right比较更稳定。

例如：
  旋转前：[0, 1, 2, 3, 4, 5, 6]
  旋转后：[4, 5, 6, 0, 1, 2, 3]
          左=(4), 中=(0), 右=(3)
  中<右 → 最小值在左侧（包含mid）
```

```java
public class FindMinRotatedArray {
    public int findMin(int[] nums) {
        if (nums == null || nums.length == 0) {
            throw new IllegalArgumentException("数组不能为空");
        }

        int left = 0, right = nums.length - 1;

        while (left < right) {
            int mid = left + (right - left) / 2;

            if (nums[mid] > nums[right]) {
                // 最小值一定在右半部分（mid不可能是最小值）
                left = mid + 1;
            } else {
                // 最小值在左半部分（mid可能就是最小值）
                right = mid;
            }
        }
        // left == right 时退出，此时指向最小值
        return nums[left];
    }
}
```

**复杂度分析**：
- 时间复杂度：O(log n)，二分查找每次将搜索范围减半
- 空间复杂度：O(1)，原地查找

## 十、今日学习要点总结

### 核心要点归纳

1. **三级缓存**是Spring解决循环依赖的核心机制：`singletonObjects`（一级）→ `earlySingletonObjects`（二级）→ `singletonFactories`（三级）
2. 循环依赖的**本质问题**是"鸡生蛋、蛋生鸡"——两个Bean互相依赖对方进行初始化
3. **构造器注入无法解决**循环依赖（任何Spring版本），Setter/字段注入可以
4. **三级缓存存在的原因**是AOP代理——需要延迟到引用时才决定返回原始对象还是代理对象
5. `@Lazy` 通过注入**懒加载代理**来解决构造器循环依赖，核心是打破Bean创建时序的耦合
6. `@Transactional` 基于 **AOP代理 + TransactionManager**，核心拦截器是 `TransactionInterceptor`
7. `@Transactional` 最常见失效原因是**自调用**（`this.xxx()`），因为绕过了代理对象
8. 事务默认只回滚 `RuntimeException` 和 `Error`，checked异常需要配置 `rollbackFor`
