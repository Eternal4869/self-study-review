## 〇、Spring事务管理版本演进

Spring的事务管理核心抽象 `PlatformTransactionManager` 自Spring 2.x引入以来一直是事务管理的基础。在Spring 6.x / Spring Boot 3.x时代，事务管理的核心不变，但有一些值得关注的变化：

| 版本 | 变化 |
|------|------|
| **Spring 5.x** | 响应式事务支持（`ReactiveTransactionManager`） |
| **Spring 6.x** | AOP底层升级，默认CGLIB代理（不再需要 `proxyTargetClass = true`） |
| **Spring Boot 3.2+** | 支持 Virtual Threads 场景下的事务管理，`@Transactional` 在虚拟线程上工作正常 |
| **Jakarta EE 9+** | 导入包从 `javax.*` 迁移到 `jakarta.*` |

**核心不变原则：**
- `@Transactional` 底层仍然是 **AOP（动态代理）**，代理机制与 Day32 循环依赖原理密切相关
- 声明式事务（`@Transactional`）仍然是主流，编程式事务（`TransactionTemplate`）用于需要精细控制的场景
- 事务传播行为7种常量完全不变

**现代选择建议：**

| 场景 | 推荐方式 |
|------|----------|
| 简单的 Service 层事务 | `@Transactional` 声明式事务 |
| 需要精细控制提交/回滚时机 | `TransactionTemplate` 编程式事务 |
| 复杂嵌套多表操作 | `@Transactional` + 正确的传播机制 |
| 响应式/WebFlux | `@Transactional` + R2DBC |

## 一、Spring事务管理核心接口

Spring事务管理基于三个核心接口，它们构成了事务管理的抽象层。

### 1.1 PlatformTransactionManager（事务管理器）

`PlatformTransactionManager` 是Spring事务管理的核心接口，所有的声明式事务和编程式事务最终都通过它来开启、提交和回滚事务。

```java
/**
 * Spring事务管理顶层接口
 * Spring 6.x: org.springframework.transaction.PlatformTransactionManager
 */
public interface PlatformTransactionManager {

    /**
     * 根据事务定义获取事务状态（开启一个新事务或加入已存在事务）
     */
    TransactionStatus getTransaction(TransactionDefinition definition)
            throws TransactionException;

    /**
     * 提交事务
     */
    void commit(TransactionStatus status) throws TransactionException;

    /**
     * 回滚事务
     */
    void rollback(TransactionStatus status) throws TransactionException;
}
```

**三个核心方法的作用：**

```
getTransaction()          commit()/rollback()
       │                        │
       ▼                        ▼
  ┌─────────┐   业务执行    ┌─────────┐
  │开启/加入│──────────────→│提交/回滚│
  │事务     │              │事务      │
  └─────────┘              └─────────┘
```

### 1.2 TransactionDefinition（事务定义）

`TransactionDefinition` 定义了事务的属性，包括传播行为、隔离级别、超时和只读标志。

```java
public interface TransactionDefinition {

    // === 传播行为常量（7种） ===
    int PROPAGATION_REQUIRED = 0;      // 默认：有则加入，无则新建
    int PROPAGATION_SUPPORTS = 1;      // 有则加入，无则非事务执行
    int PROPAGATION_MANDATORY = 2;     // 有则加入，无则抛异常
    int PROPAGATION_REQUIRES_NEW = 3;  // 总是新建，挂起当前事务
    int PROPAGATION_NOT_SUPPORTED = 4; // 非事务执行，挂起当前事务
    int PROPAGATION_NEVER = 5;         // 非事务执行，有事务则抛异常
    int PROPAGATION_NESTED = 6;        // 有则嵌套事务，无则新建

    // === 隔离级别常量 ===
    int ISOLATION_DEFAULT = -1;             // 使用数据库默认
    int ISOLATION_READ_UNCOMMITTED = 1;     // 读未提交
    int ISOLATION_READ_COMMITTED = 2;       // 读已提交
    int ISOLATION_REPEATABLE_READ = 3;      // 可重复读
    int ISOLATION_SERIALIZABLE = 4;         // 串行化

    // === 方法 ===
    int getPropagationBehavior();   // 获取传播行为
    int getIsolationLevel();        // 获取隔离级别
    int getTimeout();               // 获取超时时间（秒）
    boolean isReadOnly();           // 是否只读事务
    String getName();               // 事务名称
}
```

**事务属性总览：**

| 属性分类 | 可取值 | 默认值 | 说明 |
|----------|--------|--------|------|
| **传播行为** | 7种 | REQUIRED | 事务如何在方法间传播 |
| **隔离级别** | 4+1种 | DEFAULT | 事务并发控制 |
| **超时时间** | 秒 | -1（无限制） | 事务最长执行时间 |
| **只读标志** | true/false | false | 只读事务可做性能优化 |

### 1.3 TransactionStatus（事务状态）

`TransactionStatus` 表示当前事务的状态，提供控制事务执行和查询事务状态的方法。

```java
public interface TransactionStatus extends SavepointManager, Flushable {

    boolean isNewTransaction();      // 是否是新创建的事务
    boolean hasSavepoint();           // 是否有保存点（NESTED时会用到）
    void setRollbackOnly();           // 标记为只能回滚
    boolean isRollbackOnly();         // 是否被标记为只能回滚
    boolean isCompleted();            // 事务是否已完成

    // 继承自 SavepointManager
    Object createSavepoint();         // 创建保存点
    void rollbackToSavepoint(Object savepoint);  // 回滚到保存点
    void releaseSavepoint(Object savepoint);      // 释放保存点
}
```

### 1.4 常用实现类

| 实现类 | 适用场景 | 说明 |
|--------|----------|------|
| **DataSourceTransactionManager** | JDBC / MyBatis | 基于DataSource，最常用 |
| **JpaTransactionManager** | JPA / Hibernate | 支持JPA的EntityManager |
| **JtaTransactionManager** | 分布式事务 | 需要JTA支持 |
| **HibernateTransactionManager** | Hibernate | 已不推荐，建议用JPA |

```java
// Spring Boot自动配置示例
@Configuration
public class TxConfig {

    @Bean
    public PlatformTransactionManager transactionManager(DataSource dataSource) {
        return new DataSourceTransactionManager(dataSource);
    }
}
```

> **注意：** Spring Boot默认会自动配置事务管理器。如果引入了 `spring-boot-starter-jdbc`，会自动创建 `DataSourceTransactionManager`；如果引入了 `spring-boot-starter-data-jpa`，会自动创建 `JpaTransactionManager`。

## 二、7种事务传播机制详解

### 2.0 总览对比表

| 传播行为 | 当前存在事务 | 当前不存在事务 | 典型场景 |
|----------|-------------|---------------|----------|
| **REQUIRED** | ✅ 加入 | ✅ 新建 | 默认，绝大多数业务 |
| **REQUIRES_NEW** | ✅ 新建（挂起原） | ✅ 新建 | 日志审计 |
| **SUPPORTS** | ✅ 加入 | ❌ 非事务执行 | 纯查询 |
| **NOT_SUPPORTED** | ❌ 非事务（挂起） | ❌ 非事务 | 不需要事务的查询 |
| **MANDATORY** | ✅ 加入 | ❌ 抛异常 | 强制调用方必须有事务 |
| **NEVER** | ❌ 抛异常 | ❌ 非事务 | 强制调用方不能有事务 |
| **NESTED** | ✅ 嵌套（Savepoint） | ✅ 新建 | 有回滚点的场景 |

### 2.1 REQUIRED（默认传播行为）

**传播行为含义：** 如果当前存在事务，则加入该事务；如果当前没有事务，则创建一个新事务。

**核心特性：** 内外方法共享同一个事务，**同生共死**——一个回滚全部回滚。

**执行流程图：**

```
    调用methodB()
         │
         ▼
    ┌────────────┐
    │ 是否存在事务? │
    └──┬───────┬──┘
       │ Yes   │ No
       ▼       ▼
   ┌───────┐ ┌───────────┐
   │加入当前│ │创建新事务   │
   │事务    │ │           │
   └───┬───┘ └─────┬─────┘
       │           │
       ▼           ▼
   ┌───────────────────┐
   │  执行业务逻辑（同一 │
   │  事务上下文）      │
   └───────────────────┘
```

**代码示例：**

```java
@Service
public class OrderService {

    @Transactional(propagation = Propagation.REQUIRED)
    public void createOrder() {
        // 1. 插入订单（当前事务A）
        orderDao.insertOrder(order);

        // 2. 调用扣减库存（加入事务A）
        inventoryService.decreaseStock();  // 同生共死

        // 3. 如果扣库存失败，订单也会回滚
    }
}

@Service
public class InventoryService {

    @Transactional(propagation = Propagation.REQUIRED)
    public void decreaseStock() {
        // 加入调用方的事务，而不是创建新事务
        inventoryDao.decrease(productId, quantity);
    }
}
```

**实际场景：**

```
场景：下单减库存
  创建订单 ─┬─ REQUIRED
            │ （同一事务）
  扣减库存 ─┘

如果扣库存抛异常 → 整个事务回滚 → 订单不会创建 ✅
```

### 2.2 REQUIRES_NEW（总是新建事务）

**传播行为含义：** 总是创建一个新的事务，如果当前存在事务，则将当前事务挂起。

**核心特性：** 内外事务完全独立——各自提交、各自回滚，互不影响。

**执行流程图：**

```
    调用methodB()
         │
         ▼
    ┌────────────┐
    │ 是否存在事务? │
    └──┬───────┬──┘
       │ Yes   │ No
       ▼       ▼
   ┌───────┐  ┌────────────┐
   │挂起当前│  │直接创建     │
   │事务    │  │新事务       │
   └───┬───┘  └────────────┘
       │           │
       ▼           ▼
   ┌──────────────────────────────┐
   │  创建全新事务（独立连接/事务）│
   └──────────────────────────────┘
       │
       ▼
   ┌───────────────────────────┐
   │  methodB执行完毕 → 提交/回滚│
   │  恢复被挂起的原事务         │
   └───────────────────────────┘
```

**代码示例：**

```java
@Service
public class OrderService {

    @Transactional(propagation = Propagation.REQUIRED)
    public void createOrder() {
        // 事务A
        orderDao.insertOrder(order);      // 在事务A中执行

        // 调用日志记录
        logService.recordLog();           // 开启独立事务B

        // 如果这里抛异常，事务A回滚
        // 但日志记录已在事务B中提交，不受影响
        throw new RuntimeException("业务异常");
    }
}

@Service
public class LogService {

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void recordLog() {
        // 独立的新事务B，与事务A无关
        logDao.insert(log);  // 一定会成功写入（独立提交）
    }
}
```

**实际场景——日志审计：**

```
场景：业务操作 + 日志记录
  创建订单 ─── REQUIRED（事务A）
  记录操作日志 ─ REQUIRES_NEW（事务B，独立）

事务A回滚 → 不影响日志记录 ✅
→ 即使业务失败，操作日志也会保留（审计需要）
```

### 2.3 SUPPORTS（支持当前事务）

**传播行为含义：** 如果当前存在事务，则加入该事务；如果不存在事务，则以非事务方式执行。

```
    调用 methodB()
         │
         ▼
    ┌────────────┐
    │ 是否存在事务? │
    └──┬───────┬──┘
       │ Yes   │ No
       ▼       ▼
   ┌───────┐ ┌──────────┐
   │加入事务│ │非事务执行 │
   └───────┘ └──────────┘
```

```java
@Transactional(propagation = Propagation.SUPPORTS)
public List<User> findUsers() {
    // 有事务就用，没事务也无所谓
    // 适用场景：纯查询方法，事务可有可无
    return userDao.findAll();
}
```

---

### 2.4 NOT_SUPPORTED（不支持事务）

**传播行为含义：** 总是以非事务方式执行。如果当前存在事务，则将当前事务挂起。

```
    调用 methodB()
         │
         ▼
    ┌────────────┐
    │ 是否存在事务? │
    └──┬───────┬──┘
       │ Yes   │ No
       ▼       ▼
   ┌───────┐  ┌──────────┐
   │挂起事务│  │非事务执行 │
   └───────┘  └──────────┘
```

```java
@Transactional(propagation = Propagation.NOT_SUPPORTED)
public void exportLargeFile() {
    // 大文件导出操作耗时很长，不应该占用事务资源（数据库连接）
    // 事务耗时过长会占用连接池，影响其他请求
    fileDao.streamExport();
}
```

**典型场景：**
- 大数据量查询/导出 → 不应该占用事务连接
- 调用第三方耗时服务 → 挂起事务防止超时

### 2.5 MANDATORY（强制要求事务）

**传播行为含义：** 必须在一个已存在的事务中运行，否则抛出异常。

```
    调用 methodB()
         │
         ▼
    ┌────────────┐
    │ 是否存在事务? │
    └──┬───────┬──┘
       │ Yes   │ No
       ▼       ▼
   ┌───────┐  ┌──────────────┐
   │加入事务│  │抛异常：       │
   │       │  │No existing   │
   │       │  │transaction   │
   └───────┘  └──────────────┘
```

```java
@Transactional(propagation = Propagation.MANDATORY)
public void updateBalance(Long userId, BigDecimal amount) {
    // ❌ 如果调用方没有事务，这里直接抛异常
    // IllegalTransactionStateException
    // 用于强制调用方开启事务，避免数据不一致
    accountDao.updateBalance(userId, amount);
}
```

---

### 2.6 NEVER（禁止事务）

**传播行为含义：** 必须以非事务方式执行。如果当前存在事务，则抛出异常。

```
    调用 methodB()
         │
         ▼
    ┌────────────┐
    │ 是否存在事务? │
    └──┬───────┬──┘
       │ Yes   │ No
       ▼       ▼
   ┌──────────────┐  ┌──────────┐
   │抛异常：       │  │非事务执行 │
   │Existing      │  │           │
   │transaction   │  │           │
   └──────────────┘  └──────────┘
```

```java
@Transactional(propagation = Propagation.NEVER)
public List<User> queryForCache() {
    // 强制要求不能有事务
    // 适用于某些特定场景（如只读缓存查询，事务可能导致数据不一致）
    return redisTemplate.opsForList().range("users", 0, -1);
}
```

### 2.7 NESTED（嵌套事务）

**传播行为含义：** 与REQUIRED类似，但使用Savepoint保存点机制。如果存在事务，被调用方法创建一个嵌套事务（Savepoint）；如果不存在事务，则创建一个新事务。

**执行流程图：**

```
    调用 methodB()
         │
         ▼
    ┌────────────┐
    │ 是否存在事务? │
    └──┬───────┬──┘
       │ Yes   │ No
       ▼       ▼
   ┌───────────────┐  ┌───────────┐
   │创建Savepoint   │  │创建新事务   │
   │（嵌套事务）     │  │（同REQUIRED）│
   └───────┬───────┘  └───────────┘
           │
           ▼
   ┌───────────────────────────┐
   │  methodB执行                 │
   │  ├─ 成功 → 无需操作         │
   │  └─ 失败 → 回滚到Savepoint  │
   │          外部事务继续执行    │
   └───────────────────────────┘
```

```java
@Service
public class OrderService {

    @Transactional(propagation = Propagation.REQUIRED)
    public void createOrder() {
        // 1. 创建订单（事务A）
        orderDao.insertOrder(order);

        try {
            // 2. 创建嵌套事务（Savepoint）
            inventoryService.decreaseStock();
        } catch (Exception e) {
            // 3. 扣库存失败 → 只回滚到Savepoint，订单不回滚
            // 相当于：扣库存失败，但订单继续创建
            log.warn("扣库存失败，订单继续创建");
        }

        // 4. 外层事务最终提交
        logService.record();
        // 如果这里抛异常，上面所有操作（包括订单）都会回滚
    }
}

@Service
public class InventoryService {

    @Transactional(propagation = Propagation.NESTED)
    public void decreaseStock() {
        // 嵌套事务：失败时只回滚嵌套部分，不影响外层
        inventoryDao.decrease(productId, quantity);
        // 如果这里抛异常，回滚到嵌套的Savepoint
    }
}
```

**典型场景：** 部分失败可容忍的业务流程，如批量处理中某条失败不影响整体。

## 三、REQUIRED vs REQUIRES_NEW深入对比

### 3.1 核心区别

| 对比维度 | REQUIRED | REQUIRES_NEW |
|----------|----------|--------------|
| **事务关系** | 加入已有事务 | 总是新建 |
| **挂起外部事务？** | ❌ 不挂起 | ✅ 挂起 |
| **共用连接？** | ✅ 共用同一个DB连接 | ❌ 使用独立DB连接 |
| **回滚影响** | 一个回滚全部回滚 | 互不影响 |
| **持有锁** | 共享锁 | 独立获取锁 |

### 3.2 回滚行为对比

```
场景：methodA(REQUIRED) 调用 methodB → methodB抛异常

┌─────────────────────────────────────────────────────────┐
│  methodB传播行为        │  事务A结果  │  事务B结果        │
├─────────────────────────────────────────────────────────┤
│  REQUIRED（默认）       │  回滚      │  回滚（同一事务） │
│  REQUIRES_NEW          │  回滚      │  已提交 ✅        │
│  NESTED                │  不一定    │  回滚到Savepoint   │
└─────────────────────────────────────────────────────────┘
```

**REQUIRED回滚实验：**

```java
@Transactional(propagation = Propagation.REQUIRED)
public void methodA() {
    userDao.insert(user);           // ✅ 最终回滚
    try {
        methodB();                   // ❌ 抛异常
    } catch (RuntimeException e) {
        // 即使catch了异常也不行！
        // 因为B已标记事务为 rollbackOnly
    }
    // 提交时发现 rollbackOnly → 抛UnexpectedRollbackException
}
```

**REQUIRES_NEW回滚实验：**

```java
@Transactional(propagation = Propagation.REQUIRED)
public void methodA() {
    userDao.insert(user);           // ✅ 最终提交（独立事务）
    try {
        methodB();                   // ❌ 抛异常
    } catch (RuntimeException e) {
        // catch之后methodA正常提交 ✅
    }
}

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void methodB() {
    logDao.insert(log);
    throw new RuntimeException("methodB异常");  // B回滚
}
```

### 3.3 实际场景选择建议

**使用 REQUIRED：**
- 主业务流程（如创建订单 → 扣库存，必须全部成功）
- 业务之间有强一致性要求

**使用 REQUIRES_NEW：**

```java
// 场景1：操作日志审计——业务失败也要记录
@Service
public class AuditLogService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void recordAuditLog(String action) {
        auditLogDao.insert(new AuditLog(action));
        // 独立事务，不受主业务流程影响
    }
}

// 场景2：发送消息通知——通知失败不影响业务
@Service
public class NotificationService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void sendNotification(Long userId, String message) {
        notificationDao.insert(userId, message);
        // 独立事务，发送失败不造成主业务回滚
    }
}
```

### 3.4 完整对比示例

```java
@Service
public class CompareDemo {

    @Transactional(propagation = Propagation.REQUIRED)
    public void requiredDemo() {
        System.out.println("--- REQUIRED 演示 ---");
        userDao.insert(new User("张三"));   // 步骤1：插入用户

        try {
            requiredMethod();                // 步骤2：调用同类方法
        } catch (Exception e) {
            System.out.println("catch异常：" + e.getMessage());
        }
        // 结果：UnexpectedRollbackException
        // 原因：requiredMethod()抛异常 → 事务标记为rollbackOnly
        //       catch后继续提交 → 发现rollbackOnly → 报错
    }

    @Transactional(propagation = Propagation.REQUIRED)
    public void requiredMethod() {
        logDao.insert(new Log("日志"));
        throw new RuntimeException("内部异常");
    }

    @Transactional(propagation = Propagation.REQUIRED)
    public void requiresNewDemo() {
        System.out.println("--- REQUIRES_NEW 演示 ---");
        userDao.insert(new User("张三"));   // ✅ 步骤1：最终会提交

        try {
            requiresNewMethod();             // 步骤2：独立事务
        } catch (Exception e) {
            System.out.println("catch异常：" + e.getMessage());
        }
        // ✅ 步骤1提交成功，步骤2独立回滚
    }

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void requiresNewMethod() {
        logDao.insert(new Log("日志"));
        throw new RuntimeException("内部异常");  // 只回滚自己的事务
    }
}
```

## 四、NESTED嵌套事务原理

### 4.1 Savepoint机制

NESTED嵌套事务的实现依赖于数据库的 **Savepoint（保存点）** 机制。

```
时间线：外层事务 + NESTED嵌套事务

  BEGIN TRANSACTION          ← 外层事务开始
       │
  INSERT INTO orders ...     ← 外层操作
       │
  SAVEPOINT nested_01        ← NESTED事务开启（保存点）
       │
  UPDATE inventory ...       ← NESTED操作（扣库存）
       │
  INSERT INTO log ...        ← NESTED操作（记录日志）
       │
  ROLLBACK TO nested_01      ← NESTED失败，回滚到保存点
       │
  UPDATE order_status ...    ← 外层继续操作
       │
  COMMIT                     ← 外层提交（最终提交）
```

**代码演示Savepoint机制：**

```java
// DataSourceTransactionManager 中的核心实现逻辑（简化版）
public class NestedTransactionDemo {

    @Transactional(propagation = Propagation.REQUIRED)
    public void outerMethod() {
        // 1. 外层事务开始
        jdbcTemplate.update("INSERT INTO users(name) VALUES(?)", "张三");

        // 2. 进入NESTED → Spring内部创建Savepoint
        try {
            nestedMethod();
        } catch (Exception e) {
            // 4. NESTED失败 → Spring内部执行 rollbackToSavepoint
            // 外层事务的 users("张三") 仍然存活！
            log.warn("嵌套事务失败，外层继续");
        }

        // 5. 外层提交 → 张三被保存 ✅
        jdbcTemplate.update("UPDATE stats SET count = count + 1");
    }

    @Transactional(propagation = Propagation.NESTED)
    public void nestedMethod() {
        // 2. 此时Spring已经创建了Savepoint
        jdbcTemplate.update("INSERT INTO logs(content) VALUES(?)", "NESTED操作");
        // 3. 抛异常 → Spring执行 rollbackToSavepoint
        throw new RuntimeException("嵌套事务异常");
    }
}
```

### 4.2 NESTED回滚规则

| 情况 | 外层事务 | 嵌套事务 | 结果 |
|------|----------|----------|------|
| **嵌套事务失败（catch处理）** | ✅ 提交 | ❌ 回滚到Savepoint | 外层成功，嵌套失败 |
| **嵌套事务失败（未catch）** | ❌ 回滚 | ❌ 回滚 | 全部回滚 |
| **外层事务失败** | ❌ 回滚 | ❌ 回滚 | 全部回滚（外层回滚 → 嵌套也没了） |
| **嵌套事务成功** | ✅ 提交 | ✅ 随外层一起提交 | 全部成功 |

```java
// 关键区别：NESTED失败被catch后，外层继续提交
@Transactional
public void importantCase() {
    userDao.insert(user);  // ✅ 提交

    try {
        nestedMethod();     // NESTED失败
    } catch (Exception e) {
        // catch后外层不受影响 ✅
    }

    orderDao.insert(order); // ✅ 一起提交
}
// 结果：user + order 成功了，nestedMethod内部操作回滚了
```

### 4.3 NESTED vs REQUIRES_NEW对比

| 对比维度 | NESTED | REQUIRES_NEW |
|----------|--------|--------------|
| **实现原理** | JDBC Savepoint | 独立数据库连接 |
| **回滚独立性** | 嵌套失败可独立回滚，但外层回滚嵌套一定回滚 | 完全独立，互不影响 |
| **事务提交** | 嵌套事务随外层一起提交（等外层commit） | 自己提交自己的 |
| **数据库连接** | 共用同一个数据库连接 | 使用新数据库连接 |
| **性能** | 较好（不占用新连接） | 需要额外连接 |
| **锁竞争** | 共享锁 | 独立锁，可能死锁 |

**关键规则：**
```
NESTED: 嵌套失败 → 不影响外层
        外层失败 → 嵌套一定失败（已提交数据会一同回滚）

REQUIRES_NEW: 嵌套失败 → 不影响外层
              外层失败 → 不影响嵌套 ✅（嵌套已独立提交）
```

### 4.4 NESTED的限制

```java
// ❌ 错误：JPA/Hibernate 不支持 NESTED
@Transactional(propagation = Propagation.NESTED)
public void jpaNestedMethod() {
    entityManager.persist(entity);
    // JpaTransactionManager 不支持 Savepoint
    // 行为会上退化为 REQUIRED 或直接抛异常
}
```

| 限制 | 说明 |
|------|------|
| **仅支持JDBC** | NESTED基于JDBC的Savepoint，`DataSourceTransactionManager` 才生效 |
| **JPA不支持** | JpaTransactionManager 不支持 NESTED，行为会退化为 REQUIRED |
| **只读事务无效** | 在只读事务中使用 NESTED 无意义 |
| **嵌套深度** | 理论无限制，但Savepoint深度过大会增加数据库事务日志开销 |

## 五、事务失效场景深度分析

### 5.1 自调用失效（最经典）

**问题描述：** 在同一个类中，一个没有 `@Transactional` 的方法调用有 `@Transactional` 的方法，事务会失效。

**失效原因：** `this.methodB()` 是直接方法调用，绕过了Spring的AOP代理对象。

```
  外部调用          → 代理对象 → methodA() ✅ 事务生效
  
  内部 this.methodB() → 目标对象 → methodB() ❌ 事务失效
                           ↑
                    this 指向原始对象，不是代理对象
```

```java
@Service
public class UserService {

    // ❌ 错误写法：自调用导致事务失效
    public void register(User user) {
        // this 是原始对象，不是CGLIB代理
        this.insertUser(user);  // ← 事务失效！
    }

    @Transactional
    public void insertUser(User user) {
        userDao.insert(user);
    }
}
```

**解决方案①：通过代理对象调用**

```java
@Service
public class UserService {

    @Autowired
    private ApplicationContext context;

    public void register(User user) {
        UserService proxy = context.getBean(UserService.class);  // 获取代理对象
        proxy.insertUser(user);  // ✅ 通过代理调用，事务生效
    }

    @Transactional
    public void insertUser(User user) {
        userDao.insert(user);
    }
}
```

**解决方案②：Self-injection（推荐）**

```java
@Service
public class UserService {

    @Autowired
    private UserService self;  // Spring注入的是代理对象

    public void register(User user) {
        self.insertUser(user);  // ✅ 通过注入的代理调用
    }

    @Transactional
    public void insertUser(User user) {
        userDao.insert(user);
    }
}
```

**解决方案③：拆分Service**

```java
@Service
public class UserService {

    @Autowired
    private UserWriteService userWriteService;  // 拆分为独立Service

    public void register(User user) {
        userWriteService.insertUser(user);  // ✅ 跨Service调用，事务生效
        // 这是最推荐的做法：单一职责
    }
}
```

### 5.2 非public方法

```java
@Service
public class UserService {

    // ❌ 非public方法 → 事务失效
    @Transactional
    private void insertUserPrivate() {
        userDao.insert(user);
    }

    // ❌ protected也不行
    @Transactional
    protected void insertUserProtected() {
        userDao.insert(user);
    }

    // ❌ package-private也不行
    @Transactional
    void insertUserPackage() {
        userDao.insert(user);
    }

    // ✅ 必须是public
    @Transactional
    public void insertUserPublic() {
        userDao.insert(user);
    }
}
```

**原因：** Spring AOP基于动态代理（JDK Proxy或CGLIB）。CGLIB可以代理protected方法，但Spring的事务切面默认只对public方法生效。

### 5.3 异常被catch吞掉

```java
// ❌ 错误：异常被catch吞掉，事务不会回滚
@Transactional
public void processOrder() {
    try {
        orderDao.insert(order);
        inventoryDao.decrease();  // 这里可能抛异常
    } catch (Exception e) {
        log.error("发生异常", e);
        // 没有重新抛出 → Spring不知道有异常 → 事务提交 ❌
    }
}

// ✅ 正确：重新抛出或设置rollbackOnly
@Transactional
public void processOrder() {
    try {
        orderDao.insert(order);
        inventoryDao.decrease();
    } catch (Exception e) {
        log.error("发生异常", e);
        // 方式1：重新抛出运行时异常
        throw new RuntimeException("处理失败", e);

        // 方式2：或手动标记回滚
        // TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
    }
}
```

### 5.4 异常类型不匹配

```java
// ❌ 编译时异常（checked exception）不会触发回滚
@Transactional
public void importData(InputStream is) throws IOException {
    fileDao.importData(is);  // 抛 IOException
    // IOException 是 checked exception
    // 默认只有 RuntimeException 和 Error 才回滚
}

// ✅ 方案1：指定 rollbackFor
@Transactional(rollbackFor = Exception.class)  // 任何异常都回滚
public void importData(InputStream is) throws IOException {
    fileDao.importData(is);
}

// ✅ 方案2：指定不触发回滚的异常
@Transactional(noRollbackFor = {IllegalArgumentException.class})
public void updateWithNoRollback() {
    // 抛出 IllegalArgumentException 时不会回滚
}
```

**Spring默认回滚规则：**

| 异常类型 | 是否回滚 | 说明 |
|----------|----------|------|
| **RuntimeException** | ✅ 回滚 | 及其子类 |
| **Error** | ✅ 回滚 | 及其子类 |
| **Checked Exception** | ❌ 不回滚 | 需要手动指定 rollbackFor |

### 5.5 数据库引擎不支持

```sql
-- MySQL MyISAM引擎不支持事务！
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    ...
) ENGINE=MyISAM;  -- ❌ 事务不生效

-- 必须使用InnoDB
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    ...
) ENGINE=InnoDB;   -- ✅ 支持事务
```

### 5.6 多线程

```java
// ❌ 事务无法跨线程传播
@Transactional
public void batchProcess() {
    // 事务在main线程中
    new Thread(() -> {
        // 这里是一个新线程，没有事务！
        // ThreadLocal中的事务连接不会传递到新线程
        userDao.insert(user);
    }).start();
}
```

### 5.7 final方法

```java
// ❌ final方法不能被CGLIB代理
@Service
public class UserService {

    @Transactional
    public final void insertUser() {  // ❌ CGLIB无法重写final方法
        userDao.insert(user);
    }
}

// ❌ final类也不能被CGLIB代理
@Service
@Transactional
public final class UserService {  // ❌ 整个类的事务都不生效
    // ...
}
```

### 5.8 传播机制不当

```java
// ❌ 传播机制设置不当
@Transactional(propagation = Propagation.NEVER)
public void riskyUpdate() {
    // 设置了NEVER，但如果调用方有事务就抛异常
    // 还以为自己有事务保护，实际可能直接报错
    userDao.update(user);
}

// ❌ 跨方法事务范围过大
@Transactional
public void longRunningProcess() {
    // 事务从RPC调用到DB操作，整个过程持锁时间过长
    rpcClient.callRemoteApi();       // 耗时3000ms
    userDao.update(user);            // 耗时10ms
    // 事务持有数据库连接3000+10ms，浪费连接资源
}
```

## 六、编程式事务

### 6.1 TransactionTemplate

`TransactionTemplate` 是Spring提供的编程式事务管理工具，比声明式事务更灵活。

```java
@Service
public class ProgrammaticTxService {

    @Autowired
    private TransactionTemplate transactionTemplate;

    @Autowired
    private UserDao userDao;

    /**
     * 基本使用：有返回值
     */
    public User createUserWithResult(User user) {
        return transactionTemplate.execute(status -> {
            // 业务逻辑
            userDao.insert(user);

            // 手动设置回滚标记
            if (user.getAge() < 0) {
                status.setRollbackOnly();
                return null;
            }

            return user;
        });
    }

    /**
     * 无返回值版本
     */
    public void batchUpdate(List<User> users) {
        transactionTemplate.executeWithoutResult(status -> {
            for (User user : users) {
                try {
                    userDao.insert(user);
                } catch (Exception e) {
                    status.setRollbackOnly();  // 标记回滚
                    break;
                }
            }
        });
    }
}
```

**TransactionTemplate配置：**

```java
@Configuration
public class TxTemplateConfig {

    @Bean
    public TransactionTemplate transactionTemplate(
            PlatformTransactionManager transactionManager) {
        TransactionTemplate template = new TransactionTemplate(transactionManager);

        // 设置传播行为
        template.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRED);
        // 设置隔离级别
        template.setIsolationLevel(TransactionDefinition.ISOLATION_READ_COMMITTED);
        // 设置超时（秒）
        template.setTimeout(30);
        // 设置只读
        template.setReadOnly(false);

        return template;
    }
}
```

**精细控制回滚：**

```java
/**
 * 部分失败不影响整体的场景
 */
public void processWithPartialFail(List<Order> orders) {
    for (Order order : orders) {
        try {
            // 每个订单独立事务
            transactionTemplate.executeWithoutResult(status -> {
                orderDao.insert(order);
                inventoryDao.decrease(order.getProductId(), order.getQty());
            });
            // ✅ 单条成功，不影响下一条
        } catch (Exception e) {
            log.error("订单 {} 处理失败", order.getId(), e);
            // ❌ 单条失败，继续处理下一条
        }
    }
}
```

### 6.2 TransactionTemplate vs @Transactional 对比

| 对比维度 | @Transactional | TransactionTemplate |
|----------|----------------|---------------------|
| **使用方式** | AOP声明式 | 手动代码调用 |
| **灵活度** | 低（方法级别） | 高（代码块级别） |
| **侵入性** | 低（注解即可） | 高（需注入Template） |
| **单元测试** | 需Spring上下文 | 容易mock |
| **事务边界** | 整个方法 | 精确到代码块 |
| **传播设置** | 方法级别固定 | 可在同一方法内切换 |
| **学习曲线** | 低 | 中 |

### 6.3 编程式事务使用场景

```java
// 场景1：事务边界需要精确控制
public void processOrder(Order order) {
    // 前处理（不需要事务）
    validateOrder(order);
    enrichOrder(order);

    // 只有这里需要事务
    transactionTemplate.executeWithoutResult(status -> {
        orderDao.insert(order);
        paymentDao.createPayment(order.getPayment());
    });

    // 后处理（不需要事务）
    sendNotification(order);
}

// 场景2：同一方法中需要不同传播行为
public void complexProcess(Order order) {
    // 主事务
    transactionTemplate.execute(status -> {
        orderDao.insert(order);

        // 内部独立事务：日志
        TransactionTemplate logTemplate = new TransactionTemplate(
                transactionManager
        );
        logTemplate.setPropagationBehavior(
                TransactionDefinition.PROPAGATION_REQUIRES_NEW
        );
        logTemplate.executeWithoutResult(s -> {
            logDao.insert(new Log("订单创建:" + order.getId()));
        });
        return null;
    });
}

// 场景3：没有Spring容器管理的类中使用事务
public class StandaloneProcessor {

    private final PlatformTransactionManager txManager;

    public StandaloneProcessor(PlatformTransactionManager txManager) {
        this.txManager = txManager;
    }

    public void process() {
        TransactionTemplate template = new TransactionTemplate(txManager);
        template.executeWithoutResult(status -> {
            // 非Spring Bean也能使用事务
        });
    }
}
```

## 七、事务隔离级别

### 7.1 数据库4种隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现方式 |
|----------|------|------------|------|----------|
| **READ UNCOMMITTED** | ❌ 会 | ❌ 会 | ❌ 会 | 无锁 |
| **READ COMMITTED** | ✅ 避免 | ❌ 会 | ❌ 会 | 快照读 |
| **REPEATABLE READ** | ✅ 避免 | ✅ 避免 | ❌ 会（部分） | MVCC |
| **SERIALIZABLE** | ✅ 避免 | ✅ 避免 | ✅ 避免 | 读写锁 |

> MySQL InnoDB 默认隔离级别：**REPEATABLE READ**，通过MVCC + Next-Key Lock在REPEATABLE READ级别下也避免了幻读。

### 7.2 Spring中配置隔离级别

```java
// 方式1：注解配置
@Transactional(isolation = Isolation.READ_COMMITTED)
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    accountDao.decrease(fromId, amount);
    accountDao.increase(toId, amount);
}

// 方式2：TransactionTemplate
transactionTemplate.setIsolationLevel(
    TransactionDefinition.ISOLATION_SERIALIZABLE
);
```

### 7.3 脏读 / 不可重复读 / 幻读

```
脏读（Dirty Read）：读了别人未提交的数据
  事务A：UPDATE balance = 100 (未提交)
  事务B：SELECT balance → 读到 100 ← 脏读！
  事务A：ROLLBACK（回滚到50）
  事务B读到的 100 是脏数据

不可重复读（Non-Repeatable Read）：同一事务内两次读结果不同
  事务A：SELECT balance → 50
  事务B：UPDATE balance = 100; COMMIT;
  事务A：SELECT balance → 100 ← 不可重复读！
  事务A读到的两次结果不同

幻读（Phantom Read）：同一事务内两次查询结果集不同
  事务A：SELECT * FROM orders WHERE amount > 100 → 3条
  事务B：INSERT INTO orders VALUES(4, 200); COMMIT;
  事务A：SELECT * FROM orders WHERE amount > 100 → 4条 ← 幻读！
  事务A读到的记录数发生了"幻影"变化
```

## 八、面试常见问题

**Q1：Spring的事务传播机制有几种？默认是什么？**

Spring定义了7种事务传播行为，定义在 `TransactionDefinition` 接口中：**REQUIRED**（默认）、SUPPORTS、MANDATORY、REQUIRES_NEW、NOT_SUPPORTED、NEVER、NESTED。默认是 `REQUIRED`——有则加入，无则新建。

---

**Q2：REQUIRED和REQUIRES_NEW有什么区别？什么时候用REQUIRES_NEW？**

核心区别：**REQUIRED共享事务**（同生共死），**REQUIRES_NEW独立事务**（互不影响）。REQUIRES_NEW会挂起原事务，创建全新独立事务。

使用场景：日志审计、发送通知——这些操作即使业务失败了也应该记录。如果使用REQUIRED，业务回滚会导致日志也被回滚。

---

**Q3：NESTED和REQUIRES_NEW有什么区别？**

- **NESTED**基于JDBC **Savepoint**机制，和外层事务**共用同一个数据库连接**。外层回滚时嵌套一定回滚，嵌套失败可独立回滚。
- **REQUIRES_NEW**使用**独立的数据库连接**，完全隔离。外层回滚不影响新事务（新事务已独立提交）。
- **NESTED仅支持** `DataSourceTransactionManager`，JPA事务管理器不支持。

---

**Q4：@Transactional失效的场景有哪些？**

1. **自调用**：this.methodB() 绕过了AOP代理
2. **非public方法**：Spring AOP默认只拦截public方法
3. **异常被catch吞掉**：Spring不知道有异常
4. **异常类型不匹配**：checked exception默认不回滚，需 `rollbackFor`
5. **数据库引擎不支持**：如MyISAM
6. **多线程**：事务通过ThreadLocal传播，无法跨线程
7. **final方法/类**：CGLIB无法代理final
8. **传播机制不当**：如NEVER、NOT_SUPPORTED等

---

**Q5：为什么同一个类中方法调用事务会失效？**

Spring的事务是通过**AOP代理**实现的。外部调用时，实际调用的是**代理对象**的方法，代理对象会在方法前后添加事务管理逻辑。

但在同一个类内部用 `this.methodB()` 调用时，`this`指向的是**原始对象**（被代理的目标对象），不是代理对象，因此事务切面逻辑不会执行。

解决方案：Self-injection、AopContext.currentProxy() 或拆分Service。

---

**Q6：NESTED嵌套事务的回滚规则是什么？**

- **嵌套事务失败 + catch处理** → 嵌套事务回滚到Savepoint，外层可以继续提交
- **嵌套事务失败 + 未catch** → 异常传播到外层，整个事务一起回滚
- **外层事务失败** → 嵌套事务一定回滚（Savepoint在内层，外层回滚会覆盖）
- 核心原则：**内层回滚不影响外层，外层回滚一定影响内层**

---

**Q7：声明式事务和编程式事务怎么选择？**

90%的场景用 `@Transactional` 声明式事务即可。以下场景用编程式事务：
- 事务边界不是整个方法，只是方法内的几行代码
- 同一方法中需要不同传播行为
- 需要精细控制提交/回滚时机
- 非Spring Bean管理的普通类中需要事务

### 9.1 [61. 旋转链表（Medium）](https://leetcode.cn/problems/rotate-list/)

**题目描述：** 给定一个链表，旋转链表，将链表每个节点向右移动 `k` 个位置。

**思路分析：**

1. 先遍历计算链表长度 `len`
2. 处理 `k = k % len`，如果 `k == 0` 直接返回原链表
3. 将链表首尾相连成环
4. 找到断点：倒数第 `k` 个节点的前一个（即第 `len - k - 1` 个）
5. 断开连接，返回新头

**Java代码：**

```java
public class Solution {
    public ListNode rotateRight(ListNode head, int k) {
        // 空链表或只有一个节点，不需要旋转
        if (head == null || head.next == null || k == 0) {
            return head;
        }

        // 1. 计算链表长度，同时找到尾节点
        ListNode tail = head;
        int len = 1;
        while (tail.next != null) {
            tail = tail.next;
            len++;
        }

        // 2. 计算实际需要旋转的步数
        k = k % len;
        if (k == 0) return head;

        // 3. 成环
        tail.next = head;

        // 4. 找到断点：第 len - k 个节点就是新的尾节点
        // 需要走 len - k - 1 步到新尾节点
        ListNode newTail = head;
        for (int i = 0; i < len - k - 1; i++) {
            newTail = newTail.next;
        }

        // 5. 断开环
        ListNode newHead = newTail.next;
        newTail.next = null;

        return newHead;
    }
}
```

**复杂度分析：**
- 时间复杂度：O(n)，需要遍历链表一次计算长度，一次找断点
- 空间复杂度：O(1)，只使用了常数个额外变量

### 9.2 [82. 删除排序链表中的重复元素 II（Medium）](https://leetcode.cn/problems/remove-duplicates-from-list-ii/)

**题目描述：** 给定一个已排序的链表，删除所有含有重复数字的节点，只保留没有重复出现的数字。

**思路分析：**

1. 使用**哑节点（dummy）**简化头节点处理
2. 使用 `prev` 指针指向当前已确认不重复的最后一个节点
3. 使用 `curr` 指针扫描，遇到重复值就跳过整个重复段
4. 只有当前节点不是重复节点时，才将 `prev` 向后移动

**Java代码：**

```java
public class Solution {
    public ListNode deleteDuplicates(ListNode head) {
        // 哑节点，简化边界处理
        ListNode dummy = new ListNode(0);
        dummy.next = head;

        ListNode prev = dummy;  // 已确认不重复的最后一个节点
        ListNode curr = head;   // 当前扫描节点

        while (curr != null) {
            // 检查当前节点是否有重复
            if (curr.next != null && curr.val == curr.next.val) {
                // 跳过所有值相同的节点
                int duplicateVal = curr.val;
                while (curr != null && curr.val == duplicateVal) {
                    curr = curr.next;
                }
                // 当前重复段全部删除
                prev.next = curr;
            } else {
                // 当前节点不重复，加入结果
                prev = prev.next;
                curr = curr.next;
            }
        }

        return dummy.next;
    }
}
```

**复杂度分析：**
- 时间复杂度：O(n)，每个节点最多访问一次
- 空间复杂度：O(1)，只使用哑节点和指针变量

### 核心要点归纳

1. **PlatformTransactionManager** 是Spring事务核心接口，getTransaction/commit/rollback三个方法
2. **TransactionDefinition** 定义了传播行为、隔离级别、超时、只读等属性
3. **7种传播行为**核心必记：REQUIRED（默认/同生共死）、REQUIRES_NEW（独立）、NESTED（Savepoint）
4. **REQUIRED** 内外同事务，一个回滚全部回滚；**REQUIRES_NEW** 内外独立，互不影响
5. **NESTED** 基于数据库Savepoint，仅支持JDBC，外层回滚会影响嵌套
6. **事务失效Top3**：自调用绕过代理、异常被catch吞掉、非public方法
7. **编程式事务** TransactionTemplate 适合需要精细控制事务边界的场景
8. 事务传播与Day32循环依赖中的AOP代理机制紧密相关
