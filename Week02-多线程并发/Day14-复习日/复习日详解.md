### 1.1 线程基础（Day08）

#### 线程创建方式
| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **Thread类** | 简单直接 | 无法继承其他类 | 简单任务 |
| **Runnable接口** | 解耦任务与线程 | 无返回值 | 通用场景 |
| **Callable接口** | 有返回值，可抛异常 | 相对复杂 | 需要返回结果 |
| **线程池** | 资源复用，管理方便 | 配置复杂 | 生产环境 |

#### 线程状态转换
```
NEW → RUNNABLE → BLOCKED/WAITING/TIMED_WAITING → TERMINATED
```

#### sleep() vs wait()
| 对比项 | sleep() | wait() |
|--------|---------|--------|
| 所属类 | Thread类 | Object类 |
| 是否释放锁 | ❌ | ✅ |
| 使用位置 | 任何地方 | synchronized块中 |
| 唤醒方式 | 超时自动唤醒 | notify()/notifyAll() |

### 1.2 synchronized原理（Day09）

#### 锁升级过程
```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
```

#### 三种使用方式
```java
// 1. 实例方法 - 锁this
public synchronized void method1() {}

// 2. 静态方法 - 锁Class
public static synchronized void method2() {}

// 3. 代码块 - 锁指定对象
synchronized (lock) {}
```

#### 底层原理
- 基于Monitor对象
- monitorenter获取锁，monitorexit释放锁
- 可重入：同一线程可多次获取

### 1.3 volatile与CAS（Day10）

#### volatile两个作用
1. **保证可见性**：修改后其他线程立即可见
2. **禁止指令重排**：保证代码执行顺序
3. **不保证原子性**：复合操作不是线程安全

#### 内存屏障
```
volatile写：StoreStore屏障 → 写入 → StoreLoad屏障
volatile读：LoadLoad屏障 → 读取 → LoadStore屏障
```

#### CAS原理
```
CAS(期望值, 更新值) {
    if (当前值 == 期望值) {
        当前值 = 更新值;
        return true;
    }
    return false;
}
```

#### ABA问题
- 问题：值变化后又变回原值
- 解决：AtomicStampedReference（版本号）

### 1.4 ReentrantLock（Day11）

#### synchronized vs ReentrantLock
| 对比项 | synchronized | ReentrantLock |
|--------|-------------|---------------|
| 可中断 | ❌ | ✅ |
| 超时获取 | ❌ | ✅ |
| 公平锁 | ❌ | ✅ |
| Condition | 单一 | 多个 |

#### 公平锁vs非公平锁
| 对比项 | 公平锁 | 非公平锁 |
|--------|--------|----------|
| 顺序 | 按请求顺序 | 抢占式 |
| 性能 | 较低 | 较高 |

#### Condition精确唤醒
```java
ReentrantLock lock = new ReentrantLock();
Condition condA = lock.newCondition();
Condition condB = lock.newCondition();

// 精确唤醒
condA.signalAll();  // 只唤醒等待condA的线程
```

### 1.5 线程池（Day12）

#### 七大核心参数
```java
new ThreadPoolExecutor(
    corePoolSize,      // 核心线程数
    maximumPoolSize,   // 最大线程数
    keepAliveTime,     // 空闲时间
    unit,              // 时间单位
    workQueue,         // 任务队列
    threadFactory,     // 线程工厂
    handler            // 拒绝策略
)
```

#### execute()流程
```
核心线程 → 队列 → 非核心线程 → 拒绝策略
```

#### 四种拒绝策略
| 策略 | 行为 |
|------|------|
| AbortPolicy | 抛出异常 |
| CallerRunsPolicy | 调用者执行 |
| DiscardPolicy | 静默丢弃 |
| DiscardOldestPolicy | 丢弃最旧任务 |

#### 线程池大小
```
CPU密集型：CPU核数 + 1
IO密集型：CPU核数 * 2  计算公式：CPU核数 * (1 + IO等待时间/CPU执行时间)
```

### 1.6 JUC并发包（Day13）

#### CountDownLatch vs CyclicBarrier
| 对比项 | CountDownLatch | CyclicBarrier |
|--------|----------------|---------------|
| 可重用 | ❌ | ✅ |
| 触发条件 | 计数归零 | 所有线程到达 |
| 使用场景 | 等待完成 | 同步到达 |

#### Semaphore限流
```java
Semaphore semaphore = new Semaphore(10);  // 最多10个并发
semaphore.acquire();  // 获取许可
// 执行任务
semaphore.release();  // 释放许可
```

#### CompletableFuture异步编程
```java
CompletableFuture.supplyAsync(() -> "Hello")
    .thenApply(s -> s + " World")  // 转换结果
    .thenCompose(s -> ...)         // 扁平化
    .exceptionally(ex -> "默认值"); // 异常处理
```

## 二、核心面试题默写

### 2.1 线程池execute()流程（闭卷默写）

```
1. 当前线程数 < corePoolSize → 创建核心线程
2. 核心线程满，队列未满 → 入队等待
3. 队列满，当前线程数 < maximumPoolSize → 创建非核心线程
4. 以上都不满足 → 执行拒绝策略
```

### 2.2 线程状态转换图（闭卷默写）

```
NEW → start() → RUNNABLE
    → wait()/join() → WAITING
    → sleep(ms)/wait(ms) → TIMED_WAITING
    → 等待synchronized → BLOCKED
    → run()结束 → TERMINATED
```

### 2.3 synchronized锁升级过程（闭卷默写）

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁

偏向锁：记录Thread ID，适合单线程
轻量级锁：CAS+Lock Record，适合交替访问
重量级锁：OS Mutex，适合并发访问
```

### 2.4 volatile和synchronized的区别（闭卷默写）

| 对比项 | volatile | synchronized |
|--------|----------|--------------|
| 原子性 | ❌ | ✅ |
| 可见性 | ✅ | ✅ |
| 有序性 | ✅ | ✅ |
| 阻塞 | ❌ | ✅ |
| 使用范围 | 变量 | 方法/代码块 |

class Solution {
    private int flag = 0;
    private final Object lock = new Object();
    
    public void methodA() throws InterruptedException {
        synchronized (lock) {
            while (flag != 0) {  // 使用while检查条件
                lock.wait();
            }
            // 执行操作
            flag = 1;
            lock.notifyAll();  // 使用notifyAll
        }
    }
}

## 四、常见问题汇总

### 4.1 高频面试题

1. **为什么wait()必须在synchronized块中调用？**
   - 因为wait()会释放锁，必须先持有锁才能释放

2. **为什么线程池要使用有界队列？**
   - 防止任务堆积导致OOM

3. **为什么阿里规范禁用Executors？**
   - FixedThreadPool和SingleThreadPool队列最大长度为Integer.MAX_VALUE
   - CachedThreadPool最大线程数为Integer.MAX_VALUE
   - 都可能导致OOM

4. **volatile能保证原子性吗？**
   - 不能，volatile只保证可见性和有序性

5. **CAS的ABA问题如何解决？**
   - 使用AtomicStampedReference，通过版本号解决
