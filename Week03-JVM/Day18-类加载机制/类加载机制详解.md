# Day18 详解 — 类加载机制

## 〇、JDK 9+模块化系统对类加载的影响（面试必知）

### 0.1 模块化系统概述

JDK 9引入了**模块化系统（Project Jigsaw）**，对类加载机制产生了重要影响。

```
JDK 8 vs JDK 9+ 类加载变化：

JDK 8（类路径）：
┌─────────────────────────────────────────────────────────┐
│  ClassPath                                              │
│  ├── rt.jar                                            │
│  ├── tools.jar                                         │
│  └── 第三方jar                                         │
└─────────────────────────────────────────────────────────┘

JDK 9+（模块路径）：
┌─────────────────────────────────────────────────────────┐
│  ModulePath                                             │
│  ├── java.base模块（核心模块）                          │
│  ├── java.sql模块                                      │
│  └── 自定义模块                                         │
└─────────────────────────────────────────────────────────┘
```

### 0.2 模块化对类加载的影响

| 影响 | 说明 |
|------|------|
| **模块化类加载器** | Platform ClassLoader替代Extension ClassLoader |
| **模块依赖** | 模块间依赖关系显式声明 |
| **强封装** | 模块内部实现默认不对外暴露 |
| **模块路径** | 模块路径替代类路径 |

### 0.3 模块信息文件（module-info.java）

```java
// 模块定义示例
module com.example.myapp {
    // 依赖其他模块
    requires java.sql;
    requires java.logging;
    
    // 导出包给其他模块使用
    exports com.example.api;
    
    // 开放包给反射访问
    opens com.example.internal;
    
    // 提供服务实现
    provides com.example.spi.ServiceProvider
        with com.example.internal.ServiceProviderImpl;
}
```

## 一、类的生命周期

### 1.1 七个阶段概述

一个类从被加载到JVM内存开始，到从内存中卸载为止，它的整个生命周期包括以下七个阶段：

```
类的生命周期：

┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
│ 加载 │ → │ 验证 │ → │ 准备 │ → │ 解析 │ → │ 初始化│ → │ 使用 │ → │ 卸载 │
└──────┘   └──────┘   └──────┘   └──────┘   └──────┘   └──────┘   └──────┘
    │          │          │          │          │          │          │
    ▼          ▼          ▼          ▼          ▼          ▼          ▼
  加载字节流  文件格式    内存分配    符号引用    执行        对象使用    内存释放
  到方法区    校验        零值初始化  → 直接引用  <clinit>
```

### 1.2 验证、准备、解析合称为"连接"

```
连接（Linking）：

┌─────────────────────────────────────────┐
│               连接                       │
│  ┌──────┐   ┌──────┐   ┌──────┐        │
│  │ 验证 │ → │ 准备 │ → │ 解析 │        │
│  └──────┘   └──────┘   └──────┘        │
└─────────────────────────────────────────┘
```

## 二、加载阶段详解

### 2.1 加载（Loading）

**作用**：通过类的全限定名获取定义此类的二进制字节流，并将这个字节流所代表的静态存储结构转化为方法区的运行时数据结构。

**加载来源**：
- **本地文件系统**：.class文件
- **JAR/WAR/EAR包**：压缩包中的.class文件
- **网络**：Applet、RMI等远程加载
- **动态代理**：运行时生成代理类的字节码
- **JSP**：转换为.class文件
- **数据库**：将类的字节码存储在数据库中

```
类加载来源：

┌─────────────────────────────────────────────────────────┐
│                    加载来源                              │
├─────────────┬─────────────┬─────────────┬───────────────┤
│  .class文件 │  JAR包      │  网络      │  动态生成     │
│             │             │             │               │
│  本地文件   │  压缩包中   │  RMI       │  CGLIB        │
│  系统加载   │  .class文件 │  Applet    │  动态代理     │
└─────────────┴─────────────┴─────────────┴───────────────┘
```

### 2.2 加载过程

```java
// 类加载示例
public class ClassLoadingDemo {
    public static void main(String[] args) throws ClassNotFoundException {
        // 1. 使用Class.forName()触发类加载
        Class<?> clazz = Class.forName("java.lang.String");
        System.out.println("类名: " + clazz.getName());
        
        // 2. 使用ClassLoader加载
        ClassLoader loader = ClassLoadingDemo.class.getClassLoader();
        Class<?> clazz2 = loader.loadClass("com.example.MyClass");
        
        // 3. 使用.class获取Class对象（不会触发初始化）
        Class<?> clazz3 = ClassLoadingDemo.class;
    }
}
```

## 三、验证阶段详解

### 3.1 验证的作用

确保Class文件的字节流符合JVM规范，不会危害JVM安全。

### 3.2 四种验证类型

| 验证类型 | 说明 | 检查内容 |
|---------|------|---------|
| **文件格式验证** | 验证字节流是否符合Class文件格式 | 魔数、版本号、常量池等 |
| **元数据验证** | 语义分析，确保符合Java语言规范 | 是否有父类、是否继承了final类等 |
| **字节码验证** | 对方法体进行分析 | 指令不会跳转到方法体外、类型转换有效等 |
| **符号引用验证** | 发生在解析阶段 | 符号引用是否能找到对应的类、字段、方法 |

```
验证过程：

┌─────────────────────────────────────────────────────────┐
│  1. 文件格式验证                                         │
│     - 魔数是否为0xCAFEBABE                             │
│     - 主次版本号是否在当前JVM范围内                      │
│     - 常量池中的常量类型是否有效                         │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  2. 元数据验证                                           │
│     - 是否有且仅有一个java.lang.Object作为父类           │
│     - 是否继承了不允许继承的类（final）                  │
│     - 非抽象类是否实现了所有抽象方法                     │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  3. 字节码验证                                           │
│     - 操作数栈的数据类型与指令是否匹配                   │
│     - 跳转指令不会跳转到方法体外                         │
│     - 类型转换是否有效                                   │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  4. 符号引用验证                                         │
│     - 通过全限定名是否能找到对应的类                     │
│     - 符号引用中的访问性是否可被当前类访问               │
└─────────────────────────────────────────────────────────┘
```

## 四、准备阶段详解

### 4.1 准备的作用

为**类的静态变量**分配内存，并设置**零值初始化**。

### 4.2 注意事项

- 只为静态变量分配内存，不为实例变量分配内存
- 内存分配在方法区（元空间）
- 设置的是**零值**（0、null、false等），而非代码中赋的值
- `static final` 修饰的常量会直接赋值

```
准备阶段内存分配：

代码定义：                    准备阶段分配的内存：
private static int a = 10;   → int a = 0;      （零值）
private static String s = "hello"; → String s = null;  （零值）
private static final int b = 20;  → int b = 20;  （直接赋值，因为是final）

初始化阶段才会执行赋值：
a = 10;
s = "hello";
```

### 4.3 代码示例

```java
public class PrepareDemo {
    // 静态变量：准备阶段设置零值
    private static int a = 10;
    private static String s = "hello";
    
    // final常量：准备阶段直接赋值
    private static final int b = 20;
    
    // 实例变量：准备阶段不处理
    private int c = 30;
    
    public static void main(String[] args) {
        // 准备阶段后：
        // a = 0, s = null, b = 20
        
        // 初始化阶段后：
        // a = 10, s = "hello", b = 20
        
        System.out.println("a = " + a);      // 10
        System.out.println("s = " + s);      // hello
        System.out.println("b = " + b);      // 20
    }
}
```

## 五、解析阶段详解

### 5.1 解析的作用

将**符号引用**替换为**直接引用**的过程。

### 5.2 符号引用 vs 直接引用

| 类型 | 说明 | 示例 |
|------|------|------|
| **符号引用** | 用符号描述引用目标 | 类全限定名 `java/lang/String` |
| **直接引用** | 指向目标的指针、偏移量或句柄 | 内存地址 `0x7f32a8c0` |

```
符号引用 → 直接引用：

符号引用：
┌─────────────────────────────────────────┐
│  "java/lang/String"                     │
│  "println"                              │
│  "(Ljava/lang/String;)V"               │
└─────────────────────────────────────────┘
                    ↓ 解析
┌─────────────────────────────────────────┐
│  直接引用：                              │
│  - 指向类的指针: 0x7f32a8c0             │
│  - 方法在方法区的偏移量: 0x0048          │
│  - 本地方法栈中的句柄                    │
└─────────────────────────────────────────┘
```

### 5.3 解析的时机

- **主动引用**：会立即触发解析
- **被动引用**：不会触发解析，延迟到真正使用时

## 六、初始化阶段详解

### 6.1 初始化的作用

执行类构造器 `<clinit>()` 方法，为类的静态变量赋代码中定义的值。

### 6.2 `<clinit>()` 方法特点

- 由编译器自动收集类中所有**静态变量的赋值动作**和**静态代码块**中的语句
- 按顺序执行
- 父类的 `<clinit>()` 会先于子类执行
- JVM保证 `<clinit>()` 在多线程环境中被正确加锁同步

```
clinit()执行顺序：

public class Parent {
    static int a = 1;          // 1. 父类静态变量赋值
    static {
        System.out.println("Parent");  // 2. 父类静态块
    }
}

public class Child extends Parent {
    static int b = 2;          // 3. 子类静态变量赋值
    static {
        System.out.println("Child");   // 4. 子类静态块
    }
    
    public static void main(String[] args) {
        new Child();  // 触发初始化
    }
}

输出顺序：Parent → Child
```

### 6.3 触发类初始化的6种情况（主动引用）

| 场景 | 说明 | 示例 |
|------|------|------|
| **new实例化** | 使用new创建对象 | `new MyClass()` |
| **访问静态字段** | 读取或设置静态字段 | `MyClass.staticField` |
| **调用静态方法** | 调用静态方法 | `MyClass.staticMethod()` |
| **反射调用** | 通过反射获取类信息 | `Class.forName("MyClass")` |
| **初始化子类** | 父类未初始化时，先初始化父类 | `new Child()` 会先初始化Parent |
| **main方法** | 包含main方法的类会首先初始化 | JVM启动时 |

### 6.4 不会触发类初始化的情况（被动引用）

| 场景 | 说明 | 示例 |
|------|------|------|
| **子类引用父类静态字段** | 不会触发子类初始化 | `Child.parentStaticField` |
| **通过数组定义引用** | 不会触发数组元素类的初始化 | `MyClass[] arr = new MyClass[10]` |
| **引用常量** | �量在编译阶段已放入常量池 | `MyClass.CONSTANT` |

```java
// 被动引用示例

// 场景1：子类引用父类静态字段
class Parent {
    static int a = 10;
    static { System.out.println("Parent初始化"); }
}

class Child extends Parent {
    static int b = 20;
    static { System.out.println("Child初始化"); }
}

// 只会输出"Parent初始化"，不会输出"Child初始化"
System.out.println(Child.a);

// 场景2：通过数组定义引用
// 不会触发MyClass初始化
MyClass[] arr = new MyClass[10];

// 场景3：引用常量
class MyClass {
    static final String CONSTANT = "hello";
    static { System.out.println("MyClass初始化"); }
}

// 不会输出"MyClass初始化"，因为CONSTANT在编译期已进入常量池
System.out.println(MyClass.CONSTANT);
```

## 七、双亲委派模型

### 7.1 类加载器层次结构

```
类加载器层次结构：

┌─────────────────────────────────────────────────────────┐
│                  Bootstrap ClassLoader                   │
│                     启动类加载器                          │
│  - C++实现，加载JAVA_HOME/lib下的核心类                   │
│  - 如：rt.jar（String、Object等）                        │
└─────────────────────────────────────────────────────────┘
                              ↑ 委派
┌─────────────────────────────────────────────────────────┐
│                Extension ClassLoader                     │
│                   扩展类加载器                            │
│  - Java实现，加载JAVA_HOME/lib/ext下的扩展类             │
│  - 如：javax.*包                                        │
└─────────────────────────────────────────────────────────┘
                              ↑ 委派
┌─────────────────────────────────────────────────────────┐
│               Application ClassLoader                   │
│                   应用程序类加载器                        │
│  - Java实现，加载classpath下的类                         │
│  - 我们自定义的类默认由它加载                            │
└─────────────────────────────────────────────────────────┘
                              ↑ 委派
┌─────────────────────────────────────────────────────────┐
│               Custom ClassLoader                        │
│                   自定义类加载器                          │
│  - 用户自定义，加载特殊路径或特殊处理的类                 │
│  - 如：Tomcat的WebAppClassLoader                        │
└─────────────────────────────────────────────────────────┘
```

### 7.2 双亲委派工作流程

```
双亲委派流程：

            ┌─────────────────────────────────────────┐
            │         1. 类加载请求                     │
            └─────────────────────────────────────────┘
                          │
                          ▼
            ┌─────────────────────────────────────────┐
            │  2. 先委派给父类加载器                     │
            └─────────────────────────────────────────┘
                          │
                          ▼
            ┌─────────────────────────────────────────┐
            │  3. 父类加载器继续向上委派                 │
            └─────────────────────────────────────────┘
                          │
                          ▼
            ┌─────────────────────────────────────────┐
            │  4. 到达Bootstrap ClassLoader            │
            │     尝试加载类                           │
            └─────────────────────────────────────────┘
                          │
                    ┌─────┴─────┐
                    │           │
                    ▼           ▼
              ┌─────────┐ ┌─────────┐
              │ 能加载   │ │ 不能加载│
              └─────────┘ └─────────┘
                    │           │
                    ▼           ▼
              ┌─────────┐ ┌─────────┐
              │ 返回    │ │ 委派给  │
              │ Class  │ │ 子加载器│
              └─────────┘ └─────────┘
                               │
                               ▼
                         ┌─────────┐
                         │ 子加载器│
                         │ 尝试加载│
                         └─────────┘
                               │
                         ┌─────┴─────┐
                         │           │
                         ▼           ▼
                   ┌─────────┐ ┌─────────┐
                   │ 能加载   │ │ 不能加载│
                   └─────────┘ └─────────┘
                         │           │
                         ▼           ▼
                   ┌─────────┐ ┌─────────┐
                   │ 返回    │ │ 抛出    │
                   │ Class  │ │ ClassNotFound│
                   └─────────┘ └─────────┘
```

### 7.3 双亲委派的代码实现

```java
// java.lang.ClassLoader.loadClass() 源码
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    synchronized (getClassLoadingLock(name)) {
        // 1. 检查类是否已经加载
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                // 2. 先委派给父类加载器
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    // 3. 父类为null，使用Bootstrap ClassLoader
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // 父类加载器无法加载
            }
            
            if (c == null) {
                // 4. 父类加载器无法加载，自己尝试加载
                c = findClass(name);
            }
        }
        
        if (resolve) {
            resolveClass(c);
        }
        
        return c;
    }
}
```

### 7.4 双亲委派的好处

| 好处 | 说明 |
|------|------|
| **避免类重复加载** | 父类加载过的类，子类不会再加载 |
| **保证核心API安全** | 如自定义java.lang.String不会被加载，防止篡改核心类 |
| **保证类的唯一性** | 同一个类只会被加载一次 |

```java
// 双亲委派的安全性示例

// 即使我们在classpath下创建了java.lang.String类
// 由于双亲委派，Bootstrap ClassLoader会先加载rt.jar中的String
// 我们的自定义String永远不会被加载

package java.lang;

public class String {
    public static void main(String[] args) {
        // 这个main方法永远不会被执行
        System.out.println("自定义String");
    }
}
// 运行结果：错误，找不到main方法（使用的是rt.jar中的String）
```

## 八、打破双亲委派模型

### 8.1 为什么需要打破？

某些场景下，双亲委派模型无法满足需求：

| 场景 | 问题 | 解决方案 |
|------|------|---------|
| **SPI机制** | 核心API需要加载第三方实现 | 线程上下文类加载器 |
| **Tomcat** | Web应用隔离 | 自定义WebAppClassLoader |
| **热部署** | 动态替换类 | 自定义类加载器 |
| **OSGi** | 模块化热部署 | 网状类加载 |

### 8.2 SPI（Service Provider Interface）

**问题**：JDBC的Driver接口定义在rt.jar中，由Bootstrap ClassLoader加载。但具体实现（如MySQL驱动）在classpath下，Bootstrap ClassLoader无法加载。

**解决方案**：使用**线程上下文类加载器（Thread Context ClassLoader）**

```
SPI类加载流程：

┌─────────────────────────────────────────────────────────┐
│  1. Bootstrap ClassLoader加载 DriverManager              │
│     （在rt.jar中）                                       │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  2. DriverManager需要加载第三方Driver实现                 │
│     但Bootstrap ClassLoader无法访问classpath            │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  3. 使用Thread.currentThread().getContextClassLoader() │
│     获取线程上下文类加载器（Application ClassLoader）    │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  4. 使用线程上下文类加载器加载第三方Driver               │
│     如：com.mysql.cj.jdbc.Driver                        │
└─────────────────────────────────────────────────────────┘
```

```java
// SPI源码示例：java.sql.DriverManager
// 在static块中加载SPI实现
static {
    loadInitialDrivers();
    println("JDBC DriverManager initialized");
}

private static void loadInitialDrivers() {
    // 使用线程上下文类加载器加载SPI实现
    AccessController.doPrivileged(new PrivilegedAction<Void>() {
        public Void run() {
            // 获取线程上下文类加载器
            ClassLoader cl = Thread.currentThread().getContextClassLoader();
            // 加载META-INF/services/java.sql.Driver文件中指定的类
            ServiceLoader<Driver> loadedDrivers = ServiceLoader.load(Driver.class, cl);
            // ...
        }
    });
}
```

### 8.3 Tomcat的类加载器

**问题**：Tomcat需要部署多个Web应用，每个应用可能依赖不同版本的同一个类库。

**解决方案**：每个Web应用使用独立的**WebAppClassLoader**

```
Tomcat类加载器结构：

┌─────────────────────────────────────────────────────────┐
│                  Bootstrap ClassLoader                   │
└─────────────────────────────────────────────────────────┘
                              ↑
┌─────────────────────────────────────────────────────────┐
│                Extension ClassLoader                     │
└─────────────────────────────────────────────────────────┘
                              ↑
┌─────────────────────────────────────────────────────────┐
│               Application ClassLoader                   │
└─────────────────────────────────────────────────────────┘
                              ↑
┌─────────────────────────────────────────────────────────┐
│              Common ClassLoader                         │
│         （Tomcat和所有Web应用共享）                       │
└─────────────────────────────────────────────────────────┘
                    ↑              ↑
    ┌───────────────┴──┐    ┌─────┴───────────────┐
    │                  │    │                     │
    ▼                  ▼    ▼                     ▼
```

### 8.4 自定义类加载器

```java
// 自定义类加载器示例
public class MyClassLoader extends ClassLoader {
    private String classPath;
    
    public MyClassLoader(String classPath) {
        this.classPath = classPath;
    }
    
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            // 将类名转换为文件路径
            String fileName = name.replace(".", File.separator) + ".class";
            String filePath = classPath + File.separator + fileName;
            
            // 读取class文件
            File file = new File(filePath);
            try (FileInputStream fis = new FileInputStream(file);
                 ByteArrayOutputStream baos = new ByteArrayOutputStream()) {
                byte[] buffer = new byte[1024];
                int len;
                while ((len = fis.read(buffer)) != -1) {
                    baos.write(buffer, 0, len);
                }
                
                byte[] classData = baos.toByteArray();
                // 定义类
                return defineClass(name, classData, 0, classData.length);
            }
        } catch (IOException e) {
            throw new ClassNotFoundException("类未找到: " + name, e);
        }
    }
    
    public static void main(String[] args) throws Exception {
        MyClassLoader loader = new MyClassLoader("C:\\classes");
        
        // 加载类
        Class<?> clazz = loader.loadClass("com.example.MyClass");
        System.out.println("类加载器: " + clazz.getClassLoader());
        
        // 创建实例
        Object obj = clazz.newInstance();
        System.out.println("实例: " + obj);
    }
}
```

### 8.5 打破双亲委派的方式总结

| 方式 | 说明 | 场景 |
|------|------|------|
| **重写loadClass()** | 直接重写loadClass方法，跳过父类委派 | Tomcat WebAppClassLoader |
| **线程上下文类加载器** | 使用Thread.currentThread().getContextClassLoader() | SPI（JDBC等） |

### 9.1 [200. 岛屿数量（Medium）](https://leetcode.cn/problems/number-of-islands/)

**题目描述**：
给你一个由 `'1'`（陆地）和 `'0'`（水）组成的二维网格，请你计算网格中岛屿的数量。岛屿总是被水包围，并且每座岛屿只能由水平方向和/或垂直方向上相邻的陆地连接形成。

**思路分析**：
使用DFS或BFS遍历网格，每找到一个岛屿就将所有相连的陆地标记为已访问。

```java
// DFS方式
class Solution {
    public int numIslands(char[][] grid) {
        if (grid == null || grid.length == 0) {
            return 0;
        }
        
        int count = 0;
        int rows = grid.length;
        int cols = grid[0].length;
        
        for (int i = 0; i < rows; i++) {
            for (int j = 0; j < cols; j++) {
                if (grid[i][j] == '1') {
                    count++;
                    dfs(grid, i, j);
                }
            }
        }
        
        return count;
    }
    
    private void dfs(char[][] grid, int row, int col) {
        int rows = grid.length;
        int cols = grid[0].length;
        
        // 边界检查和访问检查
        if (row < 0 || row >= rows || col < 0 || col >= cols || grid[row][col] == '0') {
            return;
        }
        
        // 标记为已访问
        grid[row][col] = '0';
        
        // 递归访问四个方向
        dfs(grid, row - 1, col);  // 上
        dfs(grid, row + 1, col);  // 下
        dfs(grid, row, col - 1);  // 左
        dfs(grid, row, col + 1);  // 右
    }
}
```

**复杂度分析**：
- 时间复杂度：O(M × N)，其中M和N分别为行数和列数
- 空间复杂度：O(M × N)，最坏情况下递归栈深度

### 9.2 [207. 课程表（Medium）](https://leetcode.cn/problems/course-schedule/)

**题目描述**：
你这个学期必须选修 `numCourses` 门课程，记为 `0` 到 `numCourses - 1`。在选修某些课程之前需要一些先修课程。先修课程按数组 `prerequisites` 给出，其中 `prerequisites[i] = [ai, bi]` 表示如果要学习课程 `ai`，必须先完成课程 `bi`。判断是否可能完成所有课程的学习。

**思路分析**：
这是一个拓扑排序问题。如果图中没有环，则可以完成所有课程。

```java
// BFS方式（拓扑排序）
class Solution {
    public boolean canFinish(int numCourses, int[][] prerequisites) {
        // 构建邻接表和入度数组
        List<List<Integer>> graph = new ArrayList<>();
        int[] inDegree = new int[numCourses];
        
        for (int i = 0; i < numCourses; i++) {
            graph.add(new ArrayList<>());
        }
        
        for (int[] pre : prerequisites) {
            int course = pre[0];
            int prerequisite = pre[1];
            graph.get(prerequisite).add(course);
            inDegree[course]++;
        }
        
        // BFS，将入度为0的节点加入队列
        Queue<Integer> queue = new LinkedList<>();
        for (int i = 0; i < numCourses; i++) {
            if (inDegree[i] == 0) {
                queue.offer(i);
            }
        }
        
        int count = 0;
        while (!queue.isEmpty()) {
            int course = queue.poll();
            count++;
            
            for (int next : graph.get(course)) {
                inDegree[next]--;
                if (inDegree[next] == 0) {
                    queue.offer(next);
                }
            }
        }
        
        return count == numCourses;
    }
}
```

**复杂度分析**：
- 时间复杂度：O(V + E)，V为课程数，E为先修课程数
- 空间复杂度：O(V + E)
