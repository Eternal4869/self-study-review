# Day16 详解 — 垃圾回收

## 一、对象存活判断

### 1.1 引用计数法

**原理**：给每个对象添加一个引用计数器，每当有一个地方引用它时，计数器加1；引用失效时，计数器减1。计数器为0的对象即为不可用。

```
引用计数法示例：

对象A ←── 引用        计数器 = 2
  │
  └──→ 对象B ←── 引用  计数器 = 1
  
对象C (无引用)         计数器 = 0  → 可回收
```

**缺点**：无法解决**循环引用**问题

```java
// 循环引用问题示例
public class CircularReferenceDemo {
    public Object instance;
    
    public static void main(String[] args) {
        CircularReferenceDemo a = new CircularReferenceDemo();
        CircularReferenceDemo b = new CircularReferenceDemo();
        
        // 循环引用
        a.instance = b;
        b.instance = a;
        
        // 断开外部引用
        a = null;
        b = null;
        
        // 此时a和b互相引用，计数器都为1
        // 但外部已经无法访问，引用计数法无法回收
        System.gc();
    }
}
```

### 1.2 可达性分析算法（JVM实际使用）

**原理**：从一组称为 **GC Roots** 的根对象出发，沿着引用链向下搜索，如果某个对象到GC Roots之间没有任何引用链相连，则该对象不可达，可以被回收。

```
可达性分析示例：

        GC Roots
       /    |    \
      ↓     ↓     ↓
    对象A  对象B  对象C
      |     |
      ↓     ↓
    对象D  对象E
              \
               ↓
              对象F ← 无法到达GC Roots → 可回收
              
    存活对象: A, B, C, D, E
    可回收对象: F
```

### 1.3 GC Roots有哪些？

| GC Root类型 | 说明 | 示例 |
|-------------|------|------|
| **虚拟机栈引用** | 栈帧中的局部变量引用的对象 | `Object obj = new Object()` |
| **静态变量** | 类的静态变量引用的对象 | `private static Object instance` |
| **常量** | 常量池中引用的对象 | `private static final String STR = "abc"` |
| **JNI引用** | Native方法引用的对象 | JNI调用中的对象引用 |
| **同步锁** | 被synchronized持有的对象 | `synchronized(obj)` 中的obj |
| **Class对象** | 类的Class对象 | `Object.class` |

```java
// GC Roots示例
public class GCRootsDemo {
    // 1. 静态变量 - 是GC Root
    private static Object staticObj = new Object();
    
    // 2. 常量 - 是GC Root
    private static final String CONSTANT = "hello";
    
    // 3. 局部变量引用 - 是GC Root
    public void method() {
        Object localObj = new Object();  // localObj是GC Root
        // 使用localObj
        System.out.println(localObj);
    }
}
```

## 二、垃圾回收算法

### 2.1 标记-清除算法（Mark-Sweep）

**原理**：首先标记出所有需要回收的对象，标记完成后统一回收。

```
标记-清除过程：

阶段1: 标记              阶段2: 清除
┌───┬───┬───┬───┐      ┌───┬───┬───┬───┐
│ X │   │ X │   │  →   │   │   │   │   │
└───┴───┴───┴───┘      └───┴───┴───┴───┘
  X = 需要回收的对象       清除后产生碎片
```

**优点**：
- 实现简单

**缺点**：
- **效率问题**：标记和清除两个过程效率都不高
- **空间碎片**：产生大量不连续的碎片，可能导致后续分配大对象时找不到连续空间而提前触发GC

### 2.2 复制算法（Copying）

**原理**：将内存分为大小相等的两块，每次只使用其中一块。当这一块用完了，就将存活的对象复制到另一块上。

```
复制算法过程：

初始状态:              复制后:
┌─────────────┐       ┌─────────────┐
│  使用区      │       │  使用区      │
│ ┌─┬─┬─┬─┐  │       │ ┌─┬─┬─┬─┐  │
│ │X│ │X│ │  │  →    │ │ │ │ │ │  │
│ └─┴─┴─┴─┘  │       │ └─┴─┴─┴─┘  │
└─────────────┘       └─────────────┘
┌─────────────┐       ┌─────────────┐
│  空闲区      │       │  空闲区      │
│             │       │ ┌─┬─┬─┬─┐  │
│             │       │ │X│ │X│ │  │
│             │       │ └─┴─┴─┴─┘  │
└─────────────┘       └─────────────┘
```

**优点**：
- 没有内存碎片
- 分配内存时只需移动堆顶指针，效率高

**缺点**：
- 可用内存缩小为原来的一半
- 存活对象较多时，复制开销大

### 2.3 标记-整理算法（Mark-Compact）

**原理**：标记后将所有存活对象向一端移动，然后清理掉边界以外的内存。

```
标记-整理过程：

阶段1: 标记              阶段2: 整理
┌───┬───┬───┬───┐      ┌───┬───┬───┬───┐
│ X │   │ X │   │  →   │ A │ B │   │   │
└───┴───┴───┴───┘      └───┴───┴───┴───┘
  X = 存活对象            存活对象移到一端
```

**优点**：
- 没有内存碎片
- 不会浪费内存

**缺点**：
- 整理过程效率较低（需要移动对象）

### 2.4 分代收集算法（Generational Collection）

**原理**：根据对象存活周期将内存分为几块（新生代、老年代），分别采用最合适的算法。

```
分代收集策略：

新生代 (Young Generation)
┌─────────────────────────────────────────┐
│  Eden  │  S0  │  S1  │                 │
│        │      │      │                 │
│  对象存活率低，使用复制算法               │
└─────────────────────────────────────────┘

老年代 (Old Generation)
┌─────────────────────────────────────────┐
│                Old Gen                  │
│                                         │
│  对象存活率高，使用标记-清除或标记-整理   │
└─────────────────────────────────────────┘
```

### 2.5 四种算法对比

| 算法 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **标记-清除** | 实现简单 | 效率低、内存碎片 | 老年代（CMS） |
| **复制算法** | 无碎片、效率高 | 内存浪费 | 新生代 |
| **标记-整理** | 无碎片、不浪费内存 | 效率较低 | 老年代 |
| **分代收集** | 综合最优 | 实现复杂 | JVM实际使用 |

## 三、Minor GC、Major GC、Full GC

### 3.1 三种GC类型对比

| GC类型 | 触发区域 | 触发条件 | 特点 |
|--------|---------|---------|------|
| **Minor GC** | 新生代 | Eden区满 | 频繁、快速 |
| **Major GC** | 老年代 | 老年代空间不足 | 较慢 |
| **Full GC** | 整个堆+方法区 | 多种条件触发 | 最慢，应尽量避免 |

### 3.2 触发条件详解

```
Minor GC触发条件：
┌─────────────────────────────────────────┐
│  Eden区空间不足                           │
│  → 触发Minor GC                         │
│  → 存活对象复制到S0/S1                   │
│  → 清空Eden区                            │
└─────────────────────────────────────────┘

Full GC触发条件：
┌─────────────────────────────────────────┐
│  1. 老年代空间不足                        │
│  2. 方法区（元空间）不足                   │
│  3. 调用System.gc()                     │
│  4. CMS GC出现Concurrent Mode Failure    │
│  5. 每次Minor GC的平均晋升大小 > 老年代剩余空间 │
└─────────────────────────────────────────┘
```

### 3.3 代码示例

```java
public class GCDemo {
    public static void main(String[] args) {
        // 强制触发GC（不建议在生产环境使用）
        System.gc();
        
        // 查看GC信息
        // JVM参数: -verbose:gc -XX:+PrintGCDetails
        
        // 模拟大量对象创建
        List<byte[]> list = new ArrayList<>();
        for (int i = 0; i < 1000; i++) {
            list.add(new byte[1024]); // 1KB
        }
    }
}
```

## 四、对象第一次回收：finalize()

### 4.1 finaliz()方法

- 对象被回收前会调用 `finalize()` 方法
- `finalize()` 只会被调用一次
- **不推荐使用**，因为执行时机不确定，且已经被标记为废弃

### 4.2 对象状态转换

```
对象状态转换：

可达状态 → 可恢复状态 → 不可达状态
   │           │           │
   │     调用finalize()    │
   │     可能重新引用      │
   │           │           │
   │           └─────┬─────┘
   │                 │
   │         如果重新引用
   │                 │
   │                 ▼
   └──────── 可达状态（复活）
                 │
                 │ 如果没有重新引用
                 ▼
            不可达状态（回收）
```

### 4.3 代码示例

```java
public class FinalizeDemo {
    private static FinalizeDemo instance = null;
    
    @Override
    protected void finalize() throws Throwable {
        super.finalize();
        System.out.println("finalize()被调用");
        // 在finalize中重新引用自身，使对象复活
        instance = this;
    }
    
    public static void main(String[] args) throws InterruptedException {
        instance = new FinalizeDemo();
        
        // 第一次：对象可以被finalize()复活
        instance = null;
        System.gc();
        Thread.sleep(500); // 等待finalize执行
        System.out.println("第一次GC后: " + (instance != null ? "存活" : "已回收"));
        
        // 第二次：finalize()不会再次调用，对象被回收
        instance = null;
        System.gc();
        Thread.sleep(500);
        System.out.println("第二次GC后: " + (instance != null ? "存活" : "已回收"));
    }
}
// 输出：
// finalize()被调用
// 第一次GC后: 存活
// 第二次GC后: 已回收
```

## 五、内存担保机制

### 5.1 什么是内存担保？

当新生代进行Minor GC时，如果Survivor区无法容纳所有存活对象，就需要将这些对象直接晋升到老年代，这个过程称为**内存担保（HandlePromotion）**。

### 5.2 担保流程

```
Minor GC内存担保流程：

┌─────────────────────────────────────────────────────────────┐
│  Eden区满，触发Minor GC                                     │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  存活对象能否放入Survivor区？                                 │
│  └─ 是: 复制到Survivor区                                    │
│  └─ 否: ↓                                                   │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  老年代剩余空间 > 晋升对象大小？                              │
│  └─ 是: 直接晋升到老年代                                    │
│  └─ 否: ↓                                                   │
└─────────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  触发Full GC                                                │
│  └─ Full GC后老年代有空间: 晋升                              │
│  └─ Full GC后老年代仍无空间: OOM                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 相关参数

```bash
# 允许担保失败
-XX:+HandlePromotionFailure  # JDK6 Update 24后默认开启

# 老年代连续可用空间 >= 新生代每次Minor GC后平均晋升大小
# 且 HandlePromotionFailure 允许担保失败
```

### 6.3 CMS废弃说明（面试高频）

**CMS（Concurrent Mark Sweep）** 已在JDK 14中被标记为废弃（JEP 363），并在后续版本中移除。

```
CMS演进历史：

     │                │                │                 │
```

### 6.4 ZGC详解（JDK21+推荐，面试高频）

**ZGC（Z Garbage Collector）** 是JDK 11引入的超低延迟垃圾收集器，JDK 21引入分代ZGC（JEP 439），JDK 23默认使用分代模式。

```
ZGC演进历史：
JDK 11 (2018)    JDK 15 (2020)    JDK 21 (2023)    JDK 23 (2024)
     │                │                │                │
     ▼                ▼                ▼                ▼
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ 实验性   │ ──→ │ 生产就绪│ ──→ │ 分代ZGC │ ──→ │ 默认分代│
│ ZGC     │     │ ZGC     │     │ (JEP439)│     │ 模式    │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
```

**ZGC核心特点**：

| 特性 | 说明 |
|------|------|
| **停顿时间** | < 1ms（不随堆大小增加） |
| **最大堆** | 支持TB级（理论16TB+） |
| **分代模式** | JDK21引入，JDK23默认 |
| **着色指针** | 使用64位指针存储元数据 |
| **读屏障** | 使用Load Barrier实现并发处理 |
| **适用场景** | 大堆、低延迟要求高的应用 |

**ZGC参数配置**：

```bash
# 启用ZGC
-XX:+UseZGC

# 分代模式（JDK21+）
-XX:+UseZGC -XX:+ZGenerational
# JDK23+只需 -XX:+UseZGC（默认分代）

# 设置最大停顿时间目标
-XX:MaxGCPauseMillis=5  # 目标5ms（默认）

# 设置堆大小
-Xms4g -Xmx16g
```

### 6.5 Shenandoah收集器简介

**Shenandoah** 是Red Hat开发的低延迟垃圾收集器，目标是将停顿时间控制在10ms以内。

**特点**：
- 与ZGC类似，都是低延迟收集器
- 使用 Brooks Pointer（转发指针）实现并发整理
- JDK 12引入，JDK 12+生产就绪

```bash
# 启用Shenandoah
-XX:+UseShenandoahGC

# 设置目标停顿时间
-XX:ShenandoahGCHeuristics=adaptive
```

### 7.1 [206. 反转链表（Easy）](https://leetcode.cn/problems/reverse-linked-list/)

**题目描述**：
给你单链表的头节点 `head`，请你反转链表，并返回反转后的链表。

**思路分析**：
使用迭代或递归方式反转链表。

```java
// 迭代方式
class Solution {
    public ListNode reverseList(ListNode head) {
        ListNode prev = null;
        ListNode curr = head;
        
        while (curr != null) {
            ListNode next = curr.next;  // 保存下一个节点
            curr.next = prev;           // 反转指针
            prev = curr;                // 移动prev
            curr = next;                // 移动curr
        }
        
        return prev;  // prev就是新的头节点
    }
}

// 递归方式
class Solution2 {
    public ListNode reverseList(ListNode head) {
        // 递归终止条件
        if (head == null || head.next == null) {
            return head;
        }
        
        // 递归反转后面的链表
        ListNode newHead = reverseList(head.next);
        
        // 反转指针
        head.next.next = head;
        head.next = null;
        
        return newHead;
    }
}
```

**复杂度分析**：
- 时间复杂度：O(n)
- 空间复杂度：迭代 O(1)，递归 O(n)

### 7.2 [141. 环形链表（Easy）](https://leetcode.cn/problems/linked-list-cycle/)

**题目描述**：
给你一个链表的头节点 `head`，判断链表中是否有环。

**思路分析**：
使用快慢指针，快指针每次走两步，慢指针每次走一步。如果有环，两者终会相遇。

```java
public class Solution {
    public boolean hasCycle(ListNode head) {
        if (head == null || head.next == null) {
            return false;
        }
        
        ListNode slow = head;      // 慢指针
        ListNode fast = head.next; // 快指针
        
        while (slow != fast) {
            if (fast == null || fast.next == null) {
                return false;  // 快指针到达链表末尾，无环
            }
            
            slow = slow.next;        // 慢指针走一步
            fast = fast.next.next;   // 快指针走两步
        }
        
        return true;  // 两者相遇，有环
    }
}
```

**复杂度分析**：
- 时间复杂度：O(n)
- 空间复杂度：O(1)

## 八、今日学习要点总结

### 核心要点归纳

1. **对象存活判断**：JVM使用可达性分析算法（非引用计数法），从GC Roots出发
2. **GC Roots**：虚拟机栈引用、静态变量、常量、JNI引用、同步锁、Class对象
3. **四种回收算法**：标记-清除、复制、标记-整理、分代收集
4. **三种GC类型**：Minor GC（新生代）、Major GC（老年代）、Full GC（整个堆）
5. **内存担保**：Survivor放不下的对象直接晋升老年代
6. **finalize()**：不推荐使用，只会调用一次，执行时机不确定
7. **收集器演进**：CMS已废弃（JDK14），G1为JDK9+默认，ZGC为JDK21+推荐
8. **ZGC特点**：停顿<1ms，支持TB级堆，JDK21+推荐使用
